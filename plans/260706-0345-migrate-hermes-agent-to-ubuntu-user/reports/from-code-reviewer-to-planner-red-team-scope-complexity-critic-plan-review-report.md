## Finding 1: Repo's canonical systemd templates (2 copies) are never updated — a routine, precedented redeploy silently reverts the migration

- **Severity:** Critical
- **Location:** Phase 6, "Blue/Green Systemd Cutover" — steps 4/6/7 edit `/etc/systemd/system/hermes.service` directly on the host; "Related Code Files" for Phase 6 lists only host paths, no repo file.
- **Flaw:** The repo ships `templates/systemd/hermes.service` (`User=hermes`, `WorkingDirectory=/home/hermes`, `ReadWritePaths=/home/hermes/...`) and a purpose-built redeploy script, `scripts/deploy-systemd-units.sh`, whose entire job is "Syncs templates/systemd/*.service from the canonical on-host clone to /etc/systemd/system/... after editing any unit template so the fix doesn't sit undeployed." This is not hypothetical — plan `260703-1738-fix-urgent-hermes-delegation-issues` Phase 2 already used this exact repo-template → `/etc/systemd/system` deploy pattern 3 days ago (`sudo install ... templates/systemd/hermes.service /etc/systemd/system/hermes.service`) to ship a real P0 fix. There is also a second canonical clone at `/opt/hermes-optimization-guide` (confirmed present on host, `deploy-systemd-units.sh`'s `GUIDE_DIR` default) whose `templates/systemd/hermes.service` also still says `User=hermes`.
- **Failure scenario:** After this migration's Phase 6 cutover, any future unit edit (e.g. a follow-up fix from the still-in-progress `260703-1738` plan, or someone running `bash scripts/deploy-systemd-units.sh` per its documented purpose) redeploys the untouched template — silently reverting `User=/WorkingDirectory=/ReadWritePaths=` back to `hermes`. Post-Phase-8 (`userdel -r hermes`), the service then fails to start entirely (no such user); pre-Phase-8 it "succeeds" by starting under a soon-to-be-deleted account, quietly undoing the whole migration.
- **Evidence:**
  - `templates/systemd/hermes.service:12,25-27,36,40,54` — `WorkingDirectory=/home/hermes`, `EnvironmentFile=-/home/hermes/.hermes/.env`, `Environment=HOME=/home/hermes`, `ReadWritePaths=/home/hermes/.hermes /home/hermes/.ccs /home/hermes/.claude /tmp`.
  - `scripts/deploy-systemd-units.sh:1-20` — "Syncs templates/systemd/*.service from the canonical on-host clone to /etc/systemd/system/... Run this after editing any unit template so the fix doesn't sit undeployed."
  - `plans/260703-1738-fix-urgent-hermes-delegation-issues/phase-02-deploy-p0-systemd-fix.md:45-51` — the exact `sudo install ... templates/systemd/hermes.service /etc/systemd/system/hermes.service && daemon-reload && restart` command, executed 2026-07-03 (marked completed).
  - `/opt/hermes-optimization-guide` confirmed present on host (`ls -la /opt/`), its `templates/systemd/hermes.service` diffed against the workspace copy — both still say `User=hermes`/`/home/hermes/...` (only a comment block and one `ReadWritePaths` entry differ between the two clones, not the user/home fields).
- **Suggested fix:** Add a step to Phase 6 (or a new Phase 6.5) that edits `templates/systemd/hermes.service`/`hermes-dashboard.service` in the repo (and syncs `/opt/hermes-optimization-guide`) to `User=ubuntu` etc., committed alongside the cutover — not left as host-only drift from the source of truth.

## Finding 2: Phase 7's core rollback-safety claim is factually contradicted by Phase 4's own destructive step

- **Severity:** Critical
- **Location:** Phase 7, "Rollback procedure" section; plan.md "Rollback Window" section. Contradicted by Phase 4, step 6.
- **Flaw:** Phase 7 states: *"This is safe at any point in the 48h window because `/home/hermes` was never modified by Phases 1-6 (only read from)."* plan.md repeats: *"Phases 1-6 are non-destructive to `/home/hermes`."* Both are false. Phase 4 step 6 runs `sudo rm -rf /home/hermes/.ccs/instances/ccs-hermes` — a real deletion under `/home/hermes` — and it runs right after Phase 4's own smoke test, well before Phase 6 (cutover) or Phase 7 (the rollback window) even begin.
- **Failure scenario:** Some unrelated problem is discovered during Phase 7's verification (e.g. a memory-recall mismatch) and the documented rollback procedure is invoked: stop ubuntu services, re-point the symlink to `/home/hermes/.hermes/hermes-agent/venv/bin/hermes`, restart `hermes.service` as the `hermes` user. That process now reads `hermes`'s own `~/.ccs/config.yaml`, which still has the `accounts.ccs-hermes` entry (Phase 4 explicitly left it in place, calling it "dead config, harmless") — but the credential instance directory it points to was deleted in Phase 4 step 6. Rolling back "restores" a service whose default CCS-routed delegation identity is now broken, contradicting the plan's own premise that rollback fully restores prior behavior.
- **Evidence:**
  - Phase 7, line 59-60: `"This is safe at any point in the 48h window because /home/hermes was never modified by Phases 1-6 (only read from)."`
  - plan.md, "Rollback Window": `"Phases 1-6 are non-destructive to /home/hermes."`
  - Phase 4, step 6: `sudo rm -rf /home/hermes/.ccs/instances/ccs-hermes` and the accompanying note: `"Leave hermes's ~/.ccs/config.yaml accounts.ccs-hermes entry as dead config (harmless, no matching instance dir)"`.
- **Suggested fix:** Either (a) defer Phase 4 step 6's deletion until after Phase 7's 48h window closes (mirrors the pattern already used for `/home/hermes` itself), or (b) correct Phase 7/plan.md's rollback-safety claim to explicitly carve out this one exception and document that a post-Phase-4 rollback has degraded (not full) delegation functionality on the hermes side.

## Finding 3: Default delegation identity (`~/.claude/.credentials.json`, bare harness) is never reconciled — Phase 3 defers it to Phase 4, Phase 4 never mentions it

- **Severity:** High
- **Location:** Phase 3, "Key Insights" table, row "Not copied ... Handled in Phase 4, different mechanism" for `~/.claude/`; Phase 4 "Consolidate CCS/Claude Identity" (title implies both, content only covers CCS).
- **Flaw:** `~/.hermes/config.yaml`'s live `delegation:` block has `default: claude-code`, and `skills/dev/coding-agent-delegate/SKILL.md:31` confirms `harness: bare` ("plain `claude` — default, works with zero extra setup") is the default harness — i.e. the primary/default delegation path authenticates via the bare `claude` CLI's own `~/.claude/.credentials.json`, not via the `ccs-hermes` CCS profile (`harness: ccs` is opt-in and separate). Phase 3 explicitly punts reconciling `~/.claude/` to Phase 4 ("different mechanism"), but Phase 4's entire scope (title, overview, steps, Related Code Files) only touches `~/.ccs/instances/ccs-hermes` and `~/.ccs/config.yaml`. `~/.claude/.credentials.json` itself is never mentioned again anywhere in Phases 4-8.
- **Failure scenario:** `ubuntu` and `hermes` each have their own distinct, non-identical `~/.claude/.credentials.json` (confirmed different file sizes/mtimes: hermes's is 470 bytes/mtime 2026-07-05 14:10, ubuntu's is 471 bytes/mtime 2026-07-05 08:13 — not a copy of each other, unlike the `ken`/`luan`/`lucas` CCS duplication the plan's research did catch). Post-cutover, every default-harness delegated `claude-code` call runs as `ubuntu` and picks up ubuntu's own personal `~/.claude/.credentials.json` instead of whatever identity hermes was using — either silently switching the bot's default delegation quota/identity to the human operator's own Claude Code account (the exact credential-conflation problem this migration says it's retiring, just in the other direction), or behaving differently if hermes's credential was scoped/different in some way that was never investigated.
- **Evidence:**
  - Live host: `stat -c "%s %Y" ~/.claude/.credentials.json` → `471 1783239224` (ubuntu); `sudo -u hermes bash -c 'stat -c "%s %Y" ~/.claude/.credentials.json'` → `470 1783260635` (hermes) — different files.
  - Phase 3, line 36: `~/.claude/, ~/.ccs/ | Handled in Phase 4, different mechanism`.
  - Phase 4 full text (overview through security considerations) — zero occurrences of `.claude` or `credentials.json`.
  - `skills/dev/coding-agent-delegate/SKILL.md:31`: `"bare = plain claude (default — works with zero extra setup) ... Opt in only after Phase 1's smoke-test gate passes"` confirming bare/`~/.claude` is the default, `ccs` is opt-in.
  - The plan's own research table (plan.md Overview) never lists a row for `~/.claude/.credentials.json` reconciliation — the only credential-duplication finding it surfaced was the CCS `ken/luan/lucas` copies, not this one; this is an unexamined blind spot, not a deliberate scope cut.
- **Suggested fix:** Add an explicit step (Phase 4 or new phase) that inspects both `~/.claude/.credentials.json` files, determines whether hermes's is a distinct bot-scoped identity worth preserving or a duplicate to retire, and states the decision — the same rigor already applied to the CCS side.

## Finding 4: 7 scripts hardcode `/home/hermes` / `hermes:hermes` and become silently wrong (not just unused) after this migration — plan only says "decide later"

- **Severity:** High
- **Location:** plan.md Overview/Dependencies ("coexists... no shared file paths until Phase 8"); Phase 8 "Next Steps" ("left as a separate decision for the user, not automated by this plan").
- **Flaw:** The plan frames the two in-progress sibling plans as merely "coexisting" with no file conflict until Phase 8. In reality, 7 files in this repo hardcode `/home/hermes` as a functional default (not just comments), and they are provisioning/maintenance tools meant to be *re-run later* (`scripts/provision-hermes-delegation/README`-style design: "small, numbered, single-job, independently re-runnable... rotating a bot account later, re-auth, etc."). None of them gets a "now obsolete for the ubuntu deployment" note anywhere in this migration plan; Phase 8's "Next Steps" only vaguely defers the decision.
- **Failure scenario:** Someone re-runs `scripts/provision-hermes-delegation/4-merge-delegation-config.sh` (or `3-ccs-reuse-bridge.sh`) after cutover, intending to update the *live* delegation config. The script hardcodes `CONFIG="/home/hermes/.hermes/config.yaml"` (line 61) / `DST_HOME="${DST_HOME:-/home/hermes/.ccs}"` (line 84). Before Phase 8, this path still exists (services just stopped) — the script "succeeds" while editing a decommissioned, inert copy that has zero effect on the live ubuntu-hosted bot, giving false confidence that a change was applied. After Phase 8 (`userdel -r hermes`), the same script fails outright referencing a deleted path/user. Similarly, `scripts/vps-bootstrap.sh`/`vps-bootstrap-oci.sh` (used for provisioning a *fresh* host) still scaffold `/home/hermes/.hermes/...` and `chown hermes:hermes` — running either against this now-migrated host reintroduces the exact `hermes` user/isolation setup this plan just eliminated.
- **Evidence:**
  - `scripts/provision-hermes-delegation/4-merge-delegation-config.sh:61`: `CONFIG="/home/hermes/.hermes/config.yaml"`.
  - `scripts/provision-hermes-delegation/3-ccs-reuse-bridge.sh:84`: `DST_HOME="${DST_HOME:-/home/hermes/.ccs}"`.
  - `scripts/vps-bootstrap.sh:145-177` and `scripts/vps-bootstrap-oci.sh:198-230`: `sudo -u hermes mkdir -p /home/hermes/.hermes/...`, `chown -R hermes:hermes /home/hermes/.hermes`, `.env` scaffold with `PATH=/home/hermes/.local/bin:...`.
  - `plans/260704-2106-bootstrap-script-delegation-provisioning-scripts/plan.md:19`: these scripts are explicitly designed to be "independently re-runnable... rotating a bot account later, re-auth, etc." — i.e. re-invocation post-migration is a designed use case, not a hypothetical misuse.
  - This migration plan's only acknowledgment: Phase 8 "Next Steps" — `"Optionally revisit ... mark them explicitly superseded/closed ... left as a separate decision for the user, not automated by this plan."`
- **Suggested fix:** State plainly in the Overview (not just Phase 8's optional footnote) that these 7 files become dead/host-mismatched for this box once cutover completes, and add a one-line guard or README note in `scripts/provision-hermes-delegation/` so a future re-run fails loudly instead of silently no-oping or reintroducing the isolated-user pattern.

## Finding 5: Phase 1's tarball backup silently races a live SQLite write with no verification step, undermining the one rollback artifact Phase 8's confirmation gate relies on

- **Severity:** Medium
- **Location:** Phase 1, step 2 (tarball) and "Risk Assessment"; Phase 8, step 1 (confirmation gate checks `tar -tzf ... && echo "tarball readable"`).
- **Flaw:** Phase 1 acknowledges the risk ("Tarball read of live SQLite files ... while the service is running could capture a mid-write snapshot") but dismisses it because "Phase 3 restores actual state from a live re-copy at cutover time, not from this tarball (this tarball is the rollback copy, not the migration source)." That reasoning only covers the forward-migration path — it does not cover Phase 8's own gate, which treats this same tarball as the final go/no-go safety net before irreversible `userdel -r hermes`, and only checks that the archive is *readable* (`tar -tzf`), not that the SQLite files inside are structurally valid (a torn write can still produce a `tar`-readable, `sqlite3`-corrupt file).
- **Failure scenario:** Phase 8 proceeds because "tarball readable" printed successfully; days later a rollback is attempted from this backup (e.g. after some unrelated future incident makes someone reach for the Phase 1 archive as their only remaining copy) and `kanban.db`/`state.db` inside it turn out to be a torn mid-write snapshot — undetected until actually needed, at the point of maximum consequence.
- **Evidence:**
  - Phase 1, "Risk Assessment": `"Tarball read of live SQLite files (kanban.db, state.db) while the service is running could capture a mid-write snapshot — acceptable here since Phase 3 restores actual state from a live re-copy..."`
  - Phase 8, step 1: only runs `tar -tzf /home/ubuntu/hermes-full-backup-260706.tar.gz > /dev/null && echo "tarball readable"` — no `sqlite3 ... "PRAGMA integrity_check"` or equivalent on the extracted db files.
- **Suggested fix:** Add a one-line `sqlite3 <extracted-db> "PRAGMA integrity_check"` (or `.dump | head`) check to Phase 8's confirmation gate, or note in Phase 1 that the tarball's DB files are best-effort/not guaranteed consistent and should not be relied on for anything beyond config/skills/memories restoration.
