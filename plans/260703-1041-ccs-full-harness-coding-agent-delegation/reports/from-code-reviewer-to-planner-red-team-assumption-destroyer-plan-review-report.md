# Red-Team Review: CCS Full-Harness Coding-Agent Delegation Plan

Reviewer: code-reviewer (Assumption Destroyer). Scope: plan.md + phase-01..04 + brainstorm source. Verification tier: STANDARD (fact-check + contract-check applied to every cited claim).

## Finding 1: Harness never actually gets provisioned for the `hermes` service user

- **Severity:** Critical
- **Location:** plan.md Overview (lines 20-24, 31-34); Phase 1 "Architecture" section (`phase-01-bootstrap-ccs-prerequisites.md:60-64`)
- **Flaw:** The plan's entire premise depends on `/home/hermes/.claude/` (global CLAUDE.md, `rules/*`, skills catalog, hooks, agent-memory) already existing and being populated on the production VPS for the `hermes` service user. Phase 1's own Architecture section states this outright: "`ccs <profile> -p "..."` ... can read `~/.claude/` (**untouched, already readable**)". Nothing in this repo provisions that directory for `hermes`.
- **Failure scenario:** A fresh production deploy (per `scripts/vps-bootstrap.sh`) creates the `hermes` user, installs CLIs, symlinks Hermes' *own* skills into `~/.hermes/skills/` (a completely different directory), and never creates or seeds `/home/hermes/.claude/`. On first CCS-routed delegation, `ccs <profile> -p` (or bare `claude -p`, see Finding 2) finds an empty/nonexistent `~/.claude/` and loads **no** CLAUDE.md, no rules, no skills catalog, no hooks — the opposite of the plan's stated goal ("inherit the full ClaudeKit harness"), and it fails silently (no error — just a plain session with no discoverable config).
- **Evidence:**
  - `grep -rn "claudekit\|\.claude/\|~/.claude" scripts/*.sh templates/**/*` → zero hits for anything that creates/seeds `.claude/` content for `hermes`.
  - `scripts/vps-bootstrap.sh:128-149` (section 7) only does `mkdir -p /home/hermes/.hermes/skills ...` and symlinks `$GUIDE_DIR/skills/*/*/` there — never touches `~/.claude/`.
  - `phase-01-bootstrap-ccs-prerequisites.md:63` explicitly assumes `~/.claude/` is "untouched, already readable" without verifying it exists on a fresh box.
  - Additionally, `templates/systemd/hermes.service:41-42` sets `ProtectHome=read-only` / `ReadWritePaths=/home/hermes/.hermes /tmp`; Phase 1 only adds `/home/hermes/.ccs` to this list — `/home/hermes/.claude` is never added. Even if an operator manually seeds it, any harness component that needs to *write* there (hook state, agent-memory persistence) fails under the hardened unit.
- **Suggested fix:** Add an explicit Phase 1 step that provisions `~/.claude/` for the `hermes` user (e.g., `ck` CLI install / copy of the operator's CK ruleset) and add it to `ReadWritePaths` if any harness component writes there. Verify on a clean VPS, not just this dev session's already-configured `~/.claude/`.

## Finding 2: Self-contradicting premise — CCS routing changes nothing about harness-loading vs the existing bare `claude -p` call

- **Severity:** Critical
- **Location:** plan.md Overview (lines 20-24 vs. 31-34, self-contradictory); `phase-03-coding-agent-delegate-skill-ccs-routing.md:20-24` (Context Links), `:44-49` (Key Insights)
- **Flaw:** The plan's Overview justifies the whole effort by claiming the *current* bare `claude -p "..."` gives "none of the ClaudeKit harness." But the plan's own "Corrections from this plan's own research" bullet says: full harness loads for **any** call unless `--bare` is passed, and "nothing here passes `--bare`." Crucially, `--bare` is a flag of the `claude` binary itself (not of `ccs`) — confirmed live: `claude --help` shows `--bare` = "skip hooks, LSP, plugin sync, attribution, auto-memory... and CLAUDE.md auto-discovery." Phase 3's own Context Links state `ccs <profile> -p` "forwards unrecognized args straight to the underlying `claude` binary." The *current* Tier-1 invocation (`claude -p "..." --allowedTools ... --output-format json`, see `skills/dev/coding-agent-delegate/SKILL.md:64,68`) **also never passes `--bare`**. By the plan's own corrected logic, the existing bare invocation should therefore *already* load the identical harness that the new `ccs`-routed invocation loads (same `~/.claude/`, same absence of `--bare`) — the only actual difference CCS introduces is credential/identity routing, which the plan itself states: "Profile choice is purely an identity/quota decision."
- **Failure scenario:** The plan ships Phase 3 believing it fixes a harness gap. It does not — the harness gap (if real) is entirely explained by Finding 1 (missing `~/.claude/` provisioning), which affects bare `claude -p` and `ccs <profile> -p` identically. If `~/.claude/` were provisioned, bare `claude -p` would *already* have full harness today, and this whole plan delivers zero net new harness behavior — only an identity/credential swap dressed up as a harness fix. Reviewers approving this plan on the stated rationale ("so delegated sessions inherit the full CK harness") are approving a premise the plan's own fact-finding disproves.
- **Evidence:** `plan.md:20-24` (original claim) vs. `plan.md:31-34` + `phase-03...md:44-49` (self-correction that undermines it); `claude --help` output (`--bare` flag description, this session); `skills/dev/coding-agent-delegate/SKILL.md:64,68` (current invocation, no `--bare`).
- **Suggested fix:** Reframe the plan's actual value proposition honestly: this is an identity/quota-routing change (delegate as a dedicated API-key identity instead of whatever credential the service's ambient `ANTHROPIC_API_KEY` currently is), not a harness-restoration fix — unless Finding 1 is separately fixed, in which case the harness benefit applies equally to both invocation styles and doesn't require CCS at all.

## Finding 3: "Unlimited concurrency, no session-slot ceiling" is asserted, not tested — and contradicted by the package's own dependency tree

- **Severity:** High
- **Location:** `phase-03-coding-agent-delegate-skill-ccs-routing.md:55-62` (Key Insights, "One CCS API profile handles unlimited concurrency")
- **Flaw:** The claim that an API profile serves "unlimited concurrent `-p` calls... only the account's API rate limit" is reasoning from CLI docs, not an executed concurrency test. `@kaitranntt/ccs`'s own published package.json (verified via `npm view @kaitranntt/ccs`) lists `proper-lockfile: ^4.1.2` as a runtime dependency — a well-known Node file-locking library — indicating CCS internally acquires file locks for at least some profile-state operations. This directly conflicts with an unqualified "no session-slot ceiling" claim; at minimum it means concurrent `ccs <same-profile> -p` invocations may serialize on profile-state I/O (settings file writes, history append, daemon socket) even though the underlying API key itself has no per-key session cap.
- **Failure scenario:** Phase 3's `--parallel=3` worked example fires 3 concurrent `ccs ccs-hermes -p ...` calls against one profile. If CCS's lockfile serializes any part of profile bootstrap/settings read per invocation, the 3 "concurrent" calls silently serialize (worse latency than advertised) or, worse, one process holds a stale lock after a crash (a classic `proper-lockfile` failure mode) and subsequent delegations hang/timeout — a production incident with no code in this repo to catch it, since Hermes' delegate_task tool has no timeout/lock-detection logic described anywhere in this plan.
- **Evidence:** `npm view @kaitranntt/ccs` output: `dependencies: ... proper-lockfile: ^4.1.2 ...`. My own quick 2-concurrent test (`ccs <same-profile> -p "hello 1"` and `"hello 2"` in parallel, fake API key) completed without an explicit "profile busy" error, but both failed at auth (fake key) before reaching a real generation — inconclusive, does not prove the "no ceiling" claim under real load.
- **Suggested fix:** Before documenting "one profile, N concurrent calls, no ceiling" as a guarantee, run an actual load test (N≥5 concurrent real `-p` calls against one API profile) against the CCS version pinned in Phase 1, and note the lockfile dependency as a known risk in Phase 3's Risk Assessment (currently absent).

## Finding 4: New `--parallel=N` invocation syntax is undeclared and inconsistent with the only existing parallel pattern in this repo

- **Severity:** High
- **Location:** `phase-03-coding-agent-delegate-skill-ccs-routing.md:137-147` (step 6); `skills/dev/coding-agent-delegate/SKILL.md:13-25` (frontmatter `parameters:` block)
- **Flaw:** `SKILL.md`'s frontmatter `parameters:` block declares exactly three parameters: `task`, `repo`, `escalate` (verified, lines 13-25). Phase 3 step 6 adds a worked example using `/delegate_code "..." repo=myorg/app --parallel=3` — a brand-new CLI-flag-style parameter never added to `parameters:` in the same phase's frontmatter edit (step 3 only adds `harness`). This skill is consumed by an LLM reading the markdown as a playbook (there is no application code in this repo that parses these invocations — confirmed, see Finding 6) — an undeclared flag with no schema entry has no defined parsing contract at all, unlike `task=`/`repo=`/`escalate=` which are at least documented as parameters.
- **Failure scenario:** The existing, only precedent for "parallel" in this guide (`part18-coding-agents.md:81-92`, "### Parallel Delegation") uses natural-language multi-line fan-out ("In parallel: 1. /claude-code ... 2. /codex ... 3. /gemini-cli ..."), not a `--parallel=N` flag. Introducing a second, syntactically different "parallel" convention in the same skill, undeclared in frontmatter, risks the delegating LLM either ignoring the flag (falls back to 1 sequential call) or inventing its own parsing (unpredictable, un-testable — this repo has no CI that can validate LLM behavior, only YAML/frontmatter linting).
- **Evidence:** `skills/dev/coding-agent-delegate/SKILL.md:13-25` (current parameters, no `parallel` key); `phase-03...md:141-145` (new `--parallel=3` example); `part18-coding-agents.md:81-92` (existing, differently-shaped parallel pattern, verified by `sed -n '58,130p'`).
- **Suggested fix:** Either declare `parallel: {type: integer, description: ...}` in frontmatter alongside `harness`, or drop the flag syntax and express the example in the same natural-language "In parallel: 1... 2... 3..." shape the rest of the guide already uses, for consistency and so the LLM has one parallel-invocation grammar to follow.

## Finding 5: Widening a hardened production service's write access to an unvetted, single-maintainer npm package holding real API keys

- **Severity:** High
- **Location:** `phase-01-bootstrap-ccs-prerequisites.md:90-93` (implement step, `npm install -g --prefix "$HOME/.local" @kaitranntt/ccs`, no version pin); Risk Assessment table (`:141`, only covers "package name/version drifts", not supply-chain trust)
- **Flaw:** `codex` (`@openai/codex`) and `gemini` (`@google/gemini-cli`) — the two existing npm-installed CLIs this plan explicitly patterns itself after — are official first-party vendor packages. `@kaitranntt/ccs` (verified via `npm view @kaitranntt/ccs`) is maintained by a single individual (`kaitranntt <kaitran.ntt@gmail.com>`), published under a personal npm scope, with an unusually high version count (1029 published versions) and no version pin in the install line. This package will hold write access under the widened `ReadWritePaths=/home/hermes/.ccs` to per-profile state including a **real production `ANTHROPIC_API_KEY`** (per Phase 2's `ccs api create --preset anthropic --api-key ...` provisioning step) on a systemd unit that is otherwise aggressively hardened (`ProtectSystem=strict`, `NoNewPrivileges=true`, restricted syscalls, etc. — `templates/systemd/hermes.service:38-70`).
- **Failure scenario:** A compromised or malicious future publish of `@kaitranntt/ccs` (single-maintainer account takeover, or a bad release) runs with the same privileges as the hardened `hermes` service and has both the production Anthropic key and unrestricted exec of `claude`/downstream tooling. No version pin means `npm install -g` always pulls latest at bootstrap/re-run time — a supply-chain risk class this hardened unit was specifically designed to minimize for exactly this kind of dependency.
- **Evidence:** `npm view @kaitranntt/ccs` (maintainer, versions, publish recency); `phase-01...md:90-93` (unpinned install command); `templates/systemd/hermes.service:38-70` (hardening posture the new dependency undermines).
- **Suggested fix:** Pin an exact version (`@kaitranntt/ccs@8.7.0`), document a review/audit step before allowing version bumps, and flag in Security Considerations (currently silent on this) that this is a new supply-chain trust addition distinct in kind from `codex`/`gemini`.

## Finding 6: `delegation.ccs_profile` and the new `harness` parameter have zero consumers anywhere in this repo — behavior is entirely LLM-interpretation-dependent, unverifiable by any test in this plan

- **Severity:** Medium
- **Location:** `phase-02-config-template-wiring.md` (adds `delegation.ccs_profile`); `phase-03-coding-agent-delegate-skill-ccs-routing.md:113-120` (adds `harness` parameter)
- **Flaw:** `grep -rn "delegation.ccs_profile\|ccs_profile" . --include="*.md" --include="*.yaml"` (excluding this plan's own files) returns zero hits — no application code, no other skill, nothing in this repo reads this field. This is consistent with the repo being a documentation/prompt guide rather than an application codebase (Hermes' actual runtime lives in `NousResearch/hermes-agent`, external to this repo) — but it means the Contract Verifier check the plan's own "Tests After" steps rely on (`validate_skills.py`, `test_validate_skills.py`) **cannot and does not** verify that the real Hermes runtime will honor `harness: ccs` / read `delegation.ccs_profile` and construct the described shell command. Those gates only check YAML frontmatter shape, not runtime behavior.
- **Failure scenario:** The plan's Success Criteria (Phase 3) list `validate_skills.py` and `test_validate_skills.py` passing as if that constitutes proof the feature works — it only proves the markdown file is well-formed. If the actual Hermes agent's LLM doesn't reliably read `~/.hermes/config.yaml`'s `delegation.ccs_profile` and translate it into the exact invocation string documented, the whole plan ships zero behavior change while all "tests" stay green.
- **Evidence:** grep above (0 hits); `.github/scripts/validate_skills.py:15-31` (only validates `REQUIRED_KEYS`/`toolsets`, never inspects `parameters` values or example command bodies, confirmed by reading the file in full).
- **Suggested fix:** State explicitly in the plan (Success Criteria or a new "Verification Gap" note) that green CI here means "well-formed skill doc," not "confirmed runtime behavior," and that a manual/live Hermes delegation test against a real `ccs_profile` is the only way to confirm this actually works.

## Finding 7: Two of the brainstorm's three unresolved/unconfirmed assumptions are silently reversed, not merely refined, without flagging the reversal for user confirmation

- **Severity:** Medium
- **Location:** `plans/reports/brainstorm-260703-1034-ccs-full-harness-delegation-skill-report.md:63-66` (assumption 2), `:67-70` (assumption 3), `:158-167` (Unresolved Questions 2 & 3) vs. `plan.md:43-48` (Corrections) and `phase-03-coding-agent-delegate-skill-ccs-routing.md:55-62` (Key Insights)
- **Flaw:** The brainstorm (never confirmed by the user — discovery attempt explicitly got no response, per its own "Discovery Attempt" section) assumed `--parallel`'s **new** capability would be multi-**instance** fan-out ("split one task across `lucas`/`ken`/`luan` concurrently to dodge single-account rate limits" — assumption 2) and that `--auto` would mean skipping Hermes' own approval gate AND forcing inner-session non-interactivity (assumption 3). The plan does not confirm these with the user; it unilaterally **replaces** assumption 2's entire design (multi-instance/multi-profile fan-out) with a different one (single API profile, unlimited concurrency, "multiple profiles are NOT required for this" — `phase-03...md:58-62`), and **drops** assumption 3's approval-skip half entirely ("No new config field needed for that half of the brainstorm's assumption 3" — `plan.md:48`), delivering neither an `--auto` flag nor any bypass-approval mechanism anywhere in the 4 phases.
- **Failure scenario:** A user who asked for "`--auto`/`--parallel` behavior" (brainstorm's literal Problem Statement ask) and expected either a fast unattended-approval lane or a rate-limit-dodging multi-account fan-out gets neither — and the plan's own "Corrections" framing makes both cuts read as settled facts rather than flagged scope changes needing sign-off, which is exactly the drift `review-audit-self-decision.md`'s rule 3 ("Guard User Decisions Against Audit/YAGNI Drift") warns against for unconfirmed-but-user-originated feature asks.
- **Evidence:** brainstorm report lines cited above (original asks + explicit "no response within the wait window"); `plan.md:43-48` and `phase-03...md:55-62` (silent replacement/drop).
- **Suggested fix:** Add an explicit "Scope changes from brainstorm, need confirmation" callout (not buried in an Overview corrections bullet) listing: (a) no `--auto` approval-skip flag is being built — confirm this is acceptable; (b) `--parallel` now means same-profile concurrency, not the originally-assumed multi-account fan-out — confirm the rate-limit-dodging use case is out of scope, not silently abandoned.
