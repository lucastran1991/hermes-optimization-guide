# Red Team Review — CCS Full-Harness Coding-Agent Delegation Plan

Reviewer: code-reviewer (Security Adversary lens). Scope: plan.md + phase-01..04.
Verification: fact-checked every cited file path/line/config-key against the live repo and, where the plan itself claims live-CLI verification (`ccs --help`, `claude --help`), reproduced that verification in this session (both binaries are present in this sandbox).

## Finding 1: `ReadWritePaths=/home/hermes/.ccs` grants write to shared, auto-executed CCS code, not just per-profile state
- **Severity:** Critical
- **Location:** Phase 1, section "Implementation Steps (TDD)" step 5 / "Risk Assessment" row 2
- **Flaw:** Phase 1 justifies widening `ReadWritePaths` to the whole `/home/hermes/.ccs` tree on the premise that "CCS keeps ALL per-profile state under `~/.ccs/`" (session history, credentials, daemon socket, job queue). That's incomplete: `~/.ccs` also holds `shared/hooks/`, `shared/plugins/`, `shared/skills/`, `shared/commands/`, `shared/agents/`, and top-level `hooks/*.cjs` and `mcp/*.cjs` — executable JS that CCS itself loads/executes on **every** `ccs <profile>` invocation, not scoped to one profile. The plan's own Risk table acknowledges "grants write access broader than strictly needed (e.g. `.ccs/shared/` plugin cache)" but rates it Low/Low via a false equivalence: "Matches the unit's existing granularity (whole `.hermes` dir...)". `.hermes` holds config/secrets (a data-exposure risk); `.ccs/shared` + `.ccs/hooks` + `.ccs/mcp` hold **executable code CCS auto-runs** (a code-execution/persistence risk) — not the same risk class.
- **Failure scenario:** A Tier-1 delegation runs `ccs ccs-hermes -p "<task>" --allowedTools "Read,Edit,Bash"` (the "full" allowlist the skill already documents for build/test tasks). If `<task>` is attacker-influenced (see Finding 2), the Bash tool inside that session can write to `/home/hermes/.ccs/hooks/<anything>.cjs` or `/home/hermes/.ccs/shared/plugins/...` — both writable under the new `ReadWritePaths` entry, both auto-loaded by CCS. The next `ccs <any-profile> -p` call under that same `$HOME` (including future, unrelated delegations) executes the planted code. This turns a single compromised task into persistent code execution embedded in the CCS toolchain itself, not just a one-shot session compromise.
- **Evidence:** Live listing of a real `~/.ccs` tree in this session:
  ```
  /home/ubuntu/.ccs/hooks/websearch-transformer.cjs
  /home/ubuntu/.ccs/mcp/image-analysis-runtime.cjs
  /home/ubuntu/.ccs/mcp/ccs-websearch-server.cjs
  /home/ubuntu/.ccs/shared/skills, /shared/commands, /shared/plugins, /shared/agents
  ```
  Plan text: phase-01-bootstrap-ccs-prerequisites.md:37-48 (claim of "per-profile state" only) and :141-142 (Risk row rating this Low/Low).
- **Suggested fix:** Scope `ReadWritePaths` to the per-profile subpath actually needed for headless dispatch (e.g. `/home/hermes/.ccs/instances/ccs-hermes`, plus whatever lock/daemon path CCS requires), not the whole tree. If CCS requires tree-wide write (daemon coordination etc.), that must be stated explicitly as a risk-accepted trade-off, not waved away by comparison to `.hermes`.

## Finding 2: Default harness escalation broadens prompt-injection blast radius for untrusted, externally-sourced task text
- **Severity:** Critical
- **Location:** Phase 3, "Security Considerations"; cross-ref Phase 1 "Security Considerations"
- **Flaw:** `coding-agent-delegate`'s own frontmatter says `when_to_use: User invokes /delegate_code <task>` (SKILL.md:2-6) and its own security note already warns "Delegated sub-sessions may get write/exec tools... Never pass writable production credentials into a sub-session" (SKILL.md:30). `task` is free text that can originate from Telegram/Discord/Kanban-card content — i.e. untrusted external input. This plan makes `harness: ccs` (full `~/.claude/CLAUDE.md` + `~/.claude/rules/*` + skills catalog + hooks + agent-memory) the **default** for every Tier-1 `claude-code` delegation, replacing a narrowly-scoped bare `claude -p` call. Phase 3's Security Considerations section claims "No new write/exec surface" — but that's true only for tool *names* (`--allowedTools` unchanged); it does not address the qualitatively larger *ambient authority* now loaded automatically: any auto-approve behavior, additional skills, or agent-memory content reachable via the global harness now applies to a session executing attacker-influenced instructions, by default, with no opt-in step.
- **Failure scenario:** An operator (or a bypassable Telegram allowlist member) submits `/delegate_code "<task containing an embedded instruction to invoke a different, more privileged skill or read agent-memory>" repo=myorg/app`. Previously (bare `claude -p`), the sub-session had no access to the CK rules/skills catalog/agent-memory at all, capping the blast radius to the explicit `--allowedTools` list. After this plan, by default, it does.
- **Evidence:** SKILL.md:2-6, :30; plan.md:20-24 (harness description); templates/config/production.yaml:258-260 (repo's own admission: "the `denylist`... does NOT see these structured tool calls — known gap, not addressed here" — i.e. the one input-scanning control that exists doesn't cover `delegate_task` payloads at all, and this plan doesn't revisit that gap even though it's widening what runs on the other side of it).
- **Suggested fix:** Either (a) keep `harness: bare` as the default and make `ccs` opt-in per-delegation with an explicit trust decision, or (b) if `ccs` stays default, add a scoped/minimal CK profile for delegated sessions (no shared agent-memory, reduced skills catalog) rather than the full human-identical harness.

## Finding 3: `--parallel=N` and `${delegation.ccs_profile}` are unverified/fabricated interface syntax with no runtime mechanism
- **Severity:** High
- **Location:** Phase 3, "Implementation Steps" steps 3, 5, 6; Phase 4 step 3
- **Flaw:** Phase 3 introduces `/delegate_code "..." repo=myorg/app --parallel=3` and `ccs "${delegation.ccs_profile}" -p ...` as if these are established, working mechanisms. Neither has any precedent in the repo:
  - Every existing `/delegate_code` example uses `key=value` args (`repo=myorg/app`, `escalate=kanban`) — SKILL.md:153-159. `--parallel=3` mixes in GNU-flag syntax never used elsewhere for this command.
  - The repo's only existing "parallel delegation" pattern is a multi-line natural-language block across *different* agent skills ("In parallel: 1. /claude-code ... 2. /codex ..." — part18-coding-agents.md:81-92), not a numeric flag on one skill.
  - `${delegation.ccs_profile}`-style config-path templating inside a shell command example appears nowhere else in the repo (`grep` across skills/ and part*.md returns zero hits).
  - The skill's own frontmatter `parameters:` (SKILL.md:13-25) lists only `task`, `repo`, `escalate` — Phase 3 adds `harness` but never adds `parallel`, so there is no declared parameter for the new flag at all.
- **Failure scenario:** `validate_skills.py`/`test_validate_skills.py` (Phase 3's own regression gate) only checks frontmatter shape (`REQUIRED_KEYS`, `ALLOWED_TOOLSETS`) — confirmed by reading `.github/scripts/validate_skills.py:16-29`, which doesn't constrain `parameters:` at all. Both gates will pass even though `--parallel` was never wired into `parameters:` and has no demonstrated parser support, giving false confidence the feature works. If Hermes' actual slash-command parser doesn't special-case `--parallel=N`, it either errors or silently appends the literal string `--parallel=3` onto the `task` text passed to `ccs <profile> -p "<task>"`, degrading "3 isolated branches" to one un-split, un-parallel delegation — directly contradicting the worked example's claimed behavior.
- **Evidence:** SKILL.md:13-25, :153-159; part18-coding-agents.md:81-92; `.github/scripts/validate_skills.py:16-29` (parameters unconstrained); `grep -rn '\${delegation'` across repo → zero hits outside this plan.
- **Suggested fix:** Add `parallel` to `parameters:` with a declared type/enum, and verify (not assume) that Hermes' actual command parser supports `--flag=value` syntax and `${config.path}` interpolation before documenting either as a worked example.

## Finding 4: Supply-chain — unpinned, personal-scope npm package now trusted with the widened credential/hook directory
- **Severity:** High
- **Location:** Phase 1, "Implementation Steps" step 3
- **Flaw:** `@kaitranntt/ccs` is a personal npm-scope package (unlike `@openai/codex`, `@google/gemini-cli` — official vendor scopes already installed by the same script). Phase 1 installs it with `npm install -g --prefix "$HOME/.local" @kaitranntt/ccs`, no version pin, no integrity/checksum check, same fail-open posture as the vendor CLIs (`|| echo "[warn] ..."`). Combined with Finding 1 (the widened `ReadWritePaths` covering credential + auto-executed hook/plugin paths), this specific dependency now sits directly in the trust chain for both credential storage and code auto-execution on the hardened `hermes` service.
- **Failure scenario:** A malicious version bump, scope takeover, or dependency-confusion attack on `@kaitranntt/ccs` runs arbitrary code as the `hermes` user at install time (bootstrap re-runs are stated to be "non-destructive... re-runnable" per vps-bootstrap.sh:27, so this executes on every re-provision, not just once) — with write access to `~/.hermes/.env` (API keys) and, post-Phase-1, `~/.ccs` (per Finding 1, including hook/plugin auto-exec paths).
- **Evidence:** phase-01-bootstrap-ccs-prerequisites.md:90-93 (install line, no pin); scripts/vps-bootstrap.sh:118-124 (existing sibling installs, same unpinned pattern, but official scopes).
- **Suggested fix:** Pin `@kaitranntt/ccs` to a specific version/digest, and document a manual verification step (maintainer identity, package age/downloads) before trusting it on a hardened production service — the existing vendor-scope precedent isn't a security justification for a personal-scope package.

## Finding 5: Documented provisioning command puts a live API key into shell history / process list
- **Severity:** High
- **Location:** Phase 2, "Implementation Steps" step 2; Phase 3 step 4
- **Flaw:** The plan's canonical one-time provisioning command is `ccs api create --preset anthropic --api-key <ANTHROPIC_API_KEY> --target claude --yes ccs-hermes`. Live `ccs api --help` (reproduced this session) confirms `--api-key <key>` is a real, and the *only*, create-time flag for supplying the key — there is no `--api-key-env`/stdin/file-based alternative flag. Passing a real secret as a CLI argument is visible via shell history and `/proc/<pid>/cmdline` / `ps aux -ww` to any co-resident user for the process lifetime. The plan treats this as a safe, documented pattern (only checking that the *docs* use a placeholder, not that the *real* command an operator will type is itself unsafe).
- **Failure scenario:** The operator follows the documented command verbatim with their real key. The key lands in `~/.bash_history` (or the shell's equivalent) and is visible to any other local user/process via `ps` during execution — on a box that (per Phase 1) also runs a hardened, internet-facing `hermes` service.
- **Evidence:** `ccs api --help` output (reproduced live, this session): `--api-key <key>  API key (create)` is the only key-supplying option listed. Also confirmed: `ccs api create` (no flags) is documented as "interactive wizard" in the same help text — an alternative the plan never considers, despite Phase 2 itself calling this step "a documented one-time manual step" (phase-02-config-template-wiring.md:49-53), meaning the interactive path was already viable.
- **Suggested fix:** Document the interactive wizard (`ccs api create`, answering prompts) as the recommended provisioning method instead of `--api-key <key>`, or at minimum add a caveat about shell-history/process-list exposure and how to mitigate it (e.g. `HISTIGNORE`, unsetting after use).

## Finding 6: Approval-gate parity claim ignores the config's own documented denylist gap, which this plan doesn't revisit despite widening what runs behind it
- **Severity:** Medium
- **Location:** Phase 2, "Security Considerations"; Phase 3, "Security Considerations"
- **Flaw:** Both phases assert that `security.approval.require_approval`'s existing `{ tool: delegate_task, actions: [dispatch] }` entry already provides identical protection whether Tier 1 shells out to bare `claude` or `ccs <profile>`. That's true narrowly (the gate fires at `dispatch`, before either code path runs) — but `templates/config/production.yaml` itself documents, next to that exact entry, that the sibling `denylist` control "is a regex match over terminal exec strings only; it does NOT see these structured tool calls — known gap, not addressed here." The plan cites the `require_approval` gate as sufficient without acknowledging that the one content-inspection control in this config has zero visibility into the `task` string that gets embedded verbatim into a shell-quoted command — a pre-existing gap this plan doesn't touch, while (per Finding 2) increasing what a compromised delegation can reach once approved.
- **Failure scenario:** An approver clicks "approve" on a `delegate_task dispatch` request without the config layer ever having scanned the task text for injection payloads (since `denylist` doesn't apply here) — the approval step checks "should this dispatch happen at all," not "is this task text safe to interpolate into a shell-quoted CLI arg."
- **Evidence:** templates/config/production.yaml:255-261 (the gap comment sits directly above the `delegate_task` entry both phases cite as sufficient).
- **Suggested fix:** At minimum, flag this as an open risk in Phase 3's Security Considerations rather than presenting the approval gate as closing the loop; consider whether the `task` string needs shell-escaping/sanitization documentation before interpolation into `ccs "<profile>" -p "<task>"`.

## Finding 7: `harness: bare`'s `--bare` flag position is unverified against real CLI arg-parsing order
- **Severity:** Medium
- **Location:** Phase 3, "Architecture" section
- **Flaw:** The diagram shows the bare-opt-out invocation as `claude -p "<task>" --allowedTools "..." --max-turns 20 --output-format json --bare` — i.e. `--bare` appended at the very end, after `-p` and its argument. Live `claude --help` (reproduced this session) lists `--bare` as a top-level mode flag ("Minimal mode: skip hooks, LSP, plugin..."), and CLIs commonly document such mode flags as needing to precede the prompt/positional argument, not follow other option flags — the plan never verifies `claude`'s actual arg parser accepts `--bare` in trailing position alongside `-p "<task>"`.
- **Failure scenario:** If `claude`'s parser doesn't accept `--bare` after a positional/`-p` argument (untested by this plan), the `harness: bare` opt-out — the parity/debugging path the plan relies on for comparison testing — silently fails to strip the harness, undermining the one documented way to verify CCS-routing correctness against a known-bare baseline.
- **Evidence:** phase-03-coding-agent-delegate-skill-ccs-routing.md:85-87 (Architecture diagram, `--bare` trailing); no test step anywhere in Phase 3's TDD steps actually invokes `claude -p ... --bare` to confirm flag-order acceptance (steps 7-9 only re-run `validate_skills.py`/`test_validate_skills.py`, which check frontmatter, not runtime CLI behavior).
- **Suggested fix:** Add a manual verification step (`claude -p "echo test" --output-format json --bare` in the documented arg order) to Phase 3's Tests-After section before shipping the example verbatim.

## Unresolved Questions

This repo has no schema/parser code to verify against (docs-only repo) — Finding 3 is based on absence of precedent, not a confirmed negative.
- Does CCS's daemon/lock coordination actually require tree-wide write under `~/.ccs`, or would a per-instance subpath suffice? Not verifiable without CCS's own source/docs (proprietary internal tool, no public URL per Phase 2's own citation note).
