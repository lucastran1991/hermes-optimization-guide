# Red-Team Review: CCS Full-Harness Coding-Agent Delegation Plan

Reviewer stance: Failure Mode Analyst (Murphy's Law). Verification tier: STANDARD.
Both Fact Checker and Contract Verifier lenses applied per finding.

## Finding 1: CCS state-isolation claim verified against the wrong profile type — empirically false for API profiles

- **Severity:** Critical
- **Location:** Phase 1, section "Key Insights" (`~line 39-44`); Phase 3, section "Key Insights" bullet 3 (`~line 55-62`)
- **Flaw:** Phase 1 claims "CCS keeps ALL per-profile state under `~/.ccs/`... verified by listing a live `~/.ccs/instances/<profile>/` tree in this session" to justify `ReadWritePaths=/home/hermes/.ccs`. The live tree it inspected (`lucas`/`ken`/`luan`) is an **account (OAuth) profile**, not the **API profile** type this plan explicitly recommends for headless use (`ccs api create --preset anthropic ...`). These are structurally different: account profiles get a full `instances/<name>/` tree (daemon, jobs, sessions, history.jsonl); API profiles do not.
- **Failure scenario:** I created a real API profile (`ccs api create --preset anthropic --api-key <dummy> --target claude --yes zz-test-review-profile`) and ran `ccs zz-test-review-profile -p "say hi" --output-format json`. Result: (a) no `~/.ccs/instances/zz-test-review-profile/` directory was ever created — `ccs api create` only writes `~/.ccs/zz-test-review-profile.settings.json` + a `profiles: {zz-test-review-profile: {type: api, settings: ...}}` entry in `config.yaml`; (b) the invocation's own JSON output shows `"memory_paths":{"auto":"/home/ubuntu/.ccs/instances/lucas/projects/-tmp/memory/"}` and plugin cache paths under `instances/lucas/...` — i.e. runtime state from an **API-profile-routed call got attributed to the machine's currently-active account profile's instance dir** (`lucas`, a human's personal profile in this dev box), not to any directory named after the API profile invoked. On a headless `hermes` service account that has **zero** account profiles (the whole reason this plan avoids `ccs auth create` — no OAuth login possible), it is unverified whether CCS falls back to a different default, errors, or writes state directly outside `~/.ccs/` (e.g. under `~/.claude/`, which `ProtectHome=read-only` still blocks and Phase 1 never widens). This directly undercuts: (1) the `ReadWritePaths` sizing rationale (right conclusion, wrong evidence — it happens to be sufficient here only because the test box has a stray "lucas" default), and (2) Phase 3's implicit assumption that concurrent `-p` calls on one API profile don't share mutable state — they may all funnel into one shared (or entirely absent) instance directory whose behavior on a fresh headless box was never tested.
- **Evidence:** live commands run this session: `ccs api create --preset anthropic --api-key sk-ant-test-dummy... --target claude --yes zz-test-review-profile` → confirmed no `instances/zz-test-review-profile/` created (`find` returned "No such file or directory"); `ccs zz-test-review-profile -p "say hi" --output-format json` → JSON field `"memory_paths":{"auto":"/home/ubuntu/.ccs/instances/lucas/projects/-tmp/memory/"}`. Plan text: `plans/260703-1041.../phase-01-bootstrap-ccs-prerequisites.md:39-44`.
- **Suggested fix:** Re-verify the state-directory model specifically for `ccs api create`-type profiles on a box with **no** pre-existing account profiles (simulate the actual `hermes` service-account condition), not by inspecting a dev box's pre-existing human account-profile directories. Confirm where state lands when no default/active account profile exists at all before finalizing the `ReadWritePaths` scope.

## Finding 2: Default-harness switch has no rollout safety net — silent hard-fail if the one-time CCS provisioning step is skipped

- **Severity:** Critical
- **Location:** Phase 3, "Overview" (`harness` default = `ccs`) and "Requirements" (`~line 66-73`); Phase 1 (bootstrap only installs the `ccs` binary, never runs `ccs api create`); Phase 2 (config ships `ccs_profile: ccs-hermes` as a placeholder requiring separate manual provisioning)
- **Flaw:** The plan changes Tier 1's claude-code branch **default** from bare `claude -p` (works with just `claude auth login`/API key, already a known prerequisite) to `ccs <profile> -p`, where `<profile>` must be manually created via `ccs api create ...` — a step none of Phase 1's bootstrap scripts, Phase 2's config template, or any doctor/healthcheck automates or verifies. If an operator runs the prescribed bootstrap script exactly as written and starts the `hermes` service, `ccs_profile: ccs-hermes` in `production.yaml` names a profile that was never created.
- **Failure scenario:** Verified live: `ccs <nonexistent-profile> -p "hi" --output-format json` exits 1 with `ERROR: Profile 'nonexistent-profile' not found` (error code E104). Every Tier-1 `claude-code` delegation would hard-fail this way on a fresh deploy that followed Phase 1/2 to the letter but hasn't yet done the separate manual `ccs api create` step — there is no automatic fallback to `harness: bare` on CCS failure; falling back is documented only as a user-chosen opt-out, not a resilience mechanism.
- **Evidence:** live test this session, `timeout 10 ccs nonexistent-profile-xyz -p "hi" --max-turns 1 --output-format json > /tmp/ccsout.txt 2>&1; echo $?` → `EXITCODE=1`, output `Profile 'nonexistent-profile-xyz' not found`. Plan citations: `phase-02-config-template-wiring.md:49-53` ("Profile creation itself... stays a documented one-time manual step... not something the config schema or bootstrap script automates").
- **Suggested fix:** Either (a) add a bootstrap/first-boot check that verifies `delegation.ccs_profile` resolves to a real profile before the service reports healthy, or (b) make the skill's Tier-1 procedure fall back to `harness: bare` automatically on a CCS "profile not found" error rather than propagating a hard failure, or (c) keep `bare` as the shipped default and let operators opt into `ccs` only after confirming provisioning.

## Finding 3: `--parallel=N` flag is undefined in the skill's parameter schema and has no specified decomposition mechanism

- **Severity:** High
- **Location:** Phase 3, "Implementation Steps" step 6 (`~line 137-147`); frontmatter `parameters:` block, `skills/dev/coding-agent-delegate/SKILL.md:13-25`
- **Flaw:** The worked example `/delegate_code "add tests for src/payments/, split into 3 subtasks" repo=myorg/app --parallel=3` introduces a `--parallel` flag in a completely different syntax family (`--flag=value`) than every other parameter in this skill, which uses `key=value` (`repo=myorg/app`, `escalate=kanban` — confirmed via `grep -n "escalate=" skills/dev/coding-agent-delegate/SKILL.md:157-158`). The plan adds a `harness` parameter to frontmatter (step 3) but never adds `parallel` — so the shipped frontmatter's `parameters:` (`task`, `repo`, `escalate`, `harness`) still doesn't declare this flag at all. Additionally, the Procedure section (steps 1-6) has no step describing *how* a single natural-language task string gets decomposed into N independent subtasks (who splits it, how boundaries are chosen, how results are merged) — the example is illustrative prose with no backing mechanism in the same file.
- **Failure scenario:** A reader/implementer treats `--parallel=3` as a real, working flag because it appears in a "worked example" next to fully-functional syntax; there is no code in this guide repo that parses it (this is a docs/guide repo, so the actual behavior lives in the closed-source Hermes agent) and no `parameters:` entry documenting it for whatever tool eventually reads the skill's schema, creating a caller/callee contract mismatch between the documented example and the declared parameter schema.
- **Evidence:** `skills/dev/coding-agent-delegate/SKILL.md:13-25` (current parameters: `task`, `repo`, `escalate`); plan step 3 (`phase-03...md:113-120`) only adds `harness`; brainstorm report itself flagged this as an open question never resolved: `plans/reports/brainstorm-260703-1034-ccs-full-harness-delegation-skill-report.md:161` ("`--parallel`: multi-instance fan-out (assumed, new) vs multi-agent-type (existing) vs both?").
- **Suggested fix:** Either add a formal `parallel` parameter to frontmatter (type/enum/default) matching the `key=value` convention already used (e.g. `parallel=3`), or explicitly scope the example as narrative-only guidance for the operator's own orchestration, not a literal `/delegate_code` invocation flag.

## Finding 4: Same-repo, same-agent-type parallel fan-out relies on branch-only isolation, not confirmed worktree isolation — file-write race risk

- **Severity:** High
- **Location:** Phase 3, "Implementation Steps" step 6 (`~line 137-147`) and "Security Considerations" (`~line 191-197`), which reuses `part18-coding-agents.md`'s "Git Hygiene When Agents Share a Workspace" section verbatim
- **Flaw:** The reused Git Hygiene section only documents `delegation.git.isolate_branches: true` / `branch_prefix` (`part18-coding-agents.md:214-223`) — a branch-naming convention, not confirmed git-worktree (separate checkout directory) isolation. You cannot have 3 different branches checked out simultaneously in one working directory; if 3 concurrent `ccs ccs-hermes -p` processes all operate against the same physical checkout of `myorg/app` while each is nominally "on its own branch," they can still race on the same files on disk regardless of which branch each eventually commits to. The **existing** "Parallel Delegation" example in `part18-coding-agents.md:81-92` sidesteps this because it fans out across *different* agent types on manifestly disjoint targets (`src/payments/` tests vs. `worker.ts` vs. `package.json`). The **new** Phase 3 example is the first case of same-agent-type, same-subtree (`src/payments/`, split into 3 subtasks) concurrent delegation — exactly the scenario that needs worktree-level isolation to be safe, and the plan never confirms or mandates it.
- **Failure scenario:** 3 concurrent `claude -p` subprocesses under `src/payments/`, each with `Edit`/`Bash` tools, running in the same on-disk directory: process A writes file X mid-edit while process B's read of the same file gets a torn/half-written view, or A's uncommitted working-tree changes get silently included in B's `git diff`/commit — data loss or corrupted diffs, discovered only after merge review.
- **Evidence:** `part18-coding-agents.md:210-225` (Git Hygiene section, `isolate_branches`/`branch_prefix` only, no `worktree` key); `part18-coding-agents.md:81-92` (existing example uses disjoint targets across different CLIs, doesn't stress-test this path); Phase 3 plan text `~line 138-147`.
- **Suggested fix:** Explicitly require `git worktree add` (separate checkout dirs) per concurrent subtask branch in the `--parallel` example, not just a differently-named branch in a shared working directory, and say so in the added prose rather than assuming the existing (branch-only) Git Hygiene section already covers it.

## Finding 7: "Verified live" concurrency claim isn't actually supported by the cited commands

- **Severity:** Medium
- **Location:** Phase 3, "Key Insights" bullet 3 (`~line 55-62`), "Verified live this session (`ccs --help`/`ccs api --help`/`claude --help`...)" citation at top of Context Links (`~line 20-24`)
- **Flaw:** The claim "one CCS API profile handles unlimited concurrency... no session-slot ceiling" is attributed to having run `ccs --help`/`ccs api --help` this session. Neither help output (confirmed by running both live) documents anything about concurrent-invocation semantics — they only list profile-creation flags/presets. The claim is actually inferential reasoning ("an API profile is just an `ANTHROPIC_API_KEY` — no session-slot ceiling") dressed as a verified fact, and per Finding 1, the one concrete behavioral test performed this review (concurrent-adjacent: repeated invocation under an API profile) revealed unexpected shared-state attribution to a default account profile's instance dir — a scenario the "help text" verification method could never have caught.
- **Failure scenario:** A future maintainer trusts the "verified live" citation at face value and doesn't independently test actual concurrent `-p` invocations before shipping the `--parallel` example, missing the shared-instance-dir behavior this review surfaced.
- **Evidence:** `ccs --help` / `ccs api --help` output (captured this session) — no concurrency-related text in either; Finding 1's `memory_paths` evidence.
- **Suggested fix:** Downgrade the phrasing from "verified" to "inferred from API-key vs. OAuth architecture" unless an actual concurrent-invocation test is run and its output attached as evidence.
