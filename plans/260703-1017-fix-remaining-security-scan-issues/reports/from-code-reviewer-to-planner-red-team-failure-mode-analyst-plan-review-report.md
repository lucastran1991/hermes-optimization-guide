## Finding 1: Phase 4's pinned SHA is the annotated-tag OBJECT hash, not the commit hash it points to

- **Severity:** Critical
- **Location:** Phase 4, section "Refactor" / "Architecture" (`phase-04-ci-action-sha-pin.md:19,42`); root cause in `research/researcher-real-hermes-schema-and-fix-verification-report.md:103`
- **Flaw:** The plan instructs pinning `gaurav-nelson/github-action-markdown-link-check` to `499c1e7f3637c131334fa8e937c45144f79d72d2`, calling it "the commit `v1` currently resolves to." It is not a commit — it is the SHA of the **annotated tag object**. The underlying commit is a different SHA.
- **Failure scenario:** `git ls-remote https://github.com/gaurav-nelson/github-action-markdown-link-check.git 'refs/tags/v1*'` (the exact command the plan prescribes, `phase-04-ci-action-sha-pin.md:74`) returns two lines for an annotated tag: `refs/tags/v1` (tag object) and `refs/tags/v1^{}` (peeled commit). The plan's own "Tests After" comment even flags this: `# One-time SHA authenticity (research already ran this 2026-07-03; annotated tag peels via ^{})` (`phase-04-ci-action-sha-pin.md:73`) — proving the authors knew about tag-peeling — yet the SHA baked into the diff is the un-peeled tag SHA anyway. Confirmed via GitHub API: `GET /repos/.../git/tags/499c1e7f...` returns `"type": "commit"` object at `"sha": "5c5dfc0ac2e225883c0e5f03a85311ec2830d368"` — i.e. 499c1e7f... is a tag wrapper, and 5c5dfc0a... is the real commit. Practical risk: Dependabot's GitHub-Actions version-update logic and most `pin-github-action`/security tooling expect a commit SHA and can mis-detect or fail to auto-bump a tag-object pin; a tag object also becomes eligible for GC if the upstream ever force-retags `v1`, since it would no longer be reachable from any live ref — the commit itself is far more likely to stay reachable (e.g. via branch history). `codeload.github.com` happened to 200 on both SHAs in this one-off test, so CI won't visibly break today, but the plan is objectively pinning the wrong object type despite demonstrating awareness of the exact pitfall.
- **Evidence:** `git ls-remote` output: `499c1e7f3637c131334fa8e937c45144f79d72d2 refs/tags/v1` and `5c5dfc0ac2e225883c0e5f03a85311ec2830d368 refs/tags/v1^{}`. GitHub API confirms `499c1e7f...` is `type: tag` wrapping commit `5c5dfc0a...`.
- **Suggested fix:** Use the peeled SHA: `uses: gaurav-nelson/github-action-markdown-link-check@5c5dfc0ac2e225883c0e5f03a85311ec2830d368 # v1`. Update Phase 4's Refactor, Tests After/Before, and Regression Gate strings accordingly, and correct the research report's mislabeling of the tag SHA as "commit."

## Finding 2: Phase 1's Regression Gate false-passes when telegram-bot.yaml's security schema is left completely empty

- **Severity:** Critical
- **Location:** Phase 1, section "Regression Gate" (`phase-01-real-security-schema-rewrite.md:151-156`)
- **Flaw:** The gate only asserts (a) the 3 files are yamllint-clean, (b) the sum of fictional-key hits across all 3 files is 0, and (c) `production.yaml` contains `^approvals:`. It never checks that `telegram-bot.yaml` (or `security-hardened.yaml`) actually gained any real key.
- **Failure scenario:** Reproduced directly: deleted `telegram-bot.yaml`'s entire `security:` block (simulating a botched merge/edit that drops the block instead of replacing it), applied a correct real-schema rewrite to `production.yaml` and `security-hardened.yaml`, then ran the exact Regression Gate command from the phase file. Output: `PHASE 1 GATE PASS`, despite `grep -c '^security:\|^approvals:\|redact_secrets\|tirith' templates/config/telegram-bot.yaml` returning `0`. This directly contradicts the phase's own Success Criteria ("All 3 files contain only real hermes-agent keys in their security-related blocks") and Todo item ("telegram-bot.yaml: replace `security:` block (`mode: smart`)") — the gate cannot catch the single most likely real failure (a partial/incomplete edit on the third file).
- **Evidence:** Reproduced live; command and result captured in this review session. Gate text: `phase-01-real-security-schema-rewrite.md:151-156`.
- **Suggested fix:** Add explicit per-file positive assertions, e.g. `grep -q '^approvals:' templates/config/security-hardened.yaml && grep -q '^approvals:' templates/config/telegram-bot.yaml && grep -q 'redact_secrets' templates/config/telegram-bot.yaml`, mirroring what "Tests After" already partially does for 2 of 3 files — but "Tests After" also never checks telegram-bot.yaml's real-key presence, so both sections need the fix.

## Finding 3: Phase 5's Regression Gate never verifies the schedule/name realignment that is the entire point of the phase

- **Severity:** Critical
- **Location:** Phase 5, section "Regression Gate" (`phase-05-cron-drift-reconciliation.md:86-91`)
- **Flaw:** The gate checks yamllint validity, `telegram_private` count == 0, the 2 reminder jobs present, and the 3 unique jobs preserved. It never checks any schedule time or the `monthly-secret-rotation`→`monthly-rotate` rename — which are the actual "drift reconciliation" the Requirements section mandates (`phase-05-cron-drift-reconciliation.md:31-35`: "Overlapping jobs: align schedule/task/name to production.yaml").
- **Failure scenario:** Reproduced directly: fixed the `notify` channel (`telegram_private`→`telegram_dm`) and appended the 2 reminder jobs, but left `weekly-mcp-audit` at 9am (should be 10am), `weekly-cost-report` at 11am (should be 9am), `weekly-dep-audit` at noon (should be 9am), and never renamed `monthly-secret-rotation`. Ran the exact Regression Gate command: `PHASE 5 GATE PASS`. The core finding (#7, "cron schedule/channel drift") this phase exists to fix would ship half-fixed (channel only) and the gate would certify it done.
- **Evidence:** Reproduced live; gate text and job list at `templates/cron/production-crons.yaml:13-34` cross-referenced against `templates/config/production.yaml:290-294`.
- **Suggested fix:** Add explicit schedule assertions, e.g. `grep -q "weekly-mcp-audit$" -A1 ... | grep -q '0 10 \* \* 1'` per job, or a small diff/normalize script comparing the two files' overlapping job schedules programmatically rather than via ad hoc grep.

## Finding 4: Phase 2's Regression Gate never verifies vps-bootstrap-oci.sh actually installs Node

- **Severity:** High
- **Location:** Phase 2, section "Regression Gate" (`phase-02-nodesource-install-hardening.md:92-97`)
- **Flaw:** The gate's two positive assertions (`grep -q 'signed-by=/usr/share/keyrings/nodesource.gpg' scripts/vps-bootstrap.sh` and `grep -q 'cloudsmith.io/public/caddy' scripts/vps-bootstrap.sh`) both target `vps-bootstrap.sh` only. The only check touching `vps-bootstrap-oci.sh` is the negative one (`setup_20.x` count == 0, summed across both files).
- **Failure scenario:** Reproduced directly: applied the correct fix to `vps-bootstrap.sh`, but replaced `vps-bootstrap-oci.sh`'s entire Node-install block with a no-op placeholder (`echo "TODO: install node here"` — no `nodesource.gpg`, no `apt-get install nodejs` at all). Ran the exact Regression Gate command: `PHASE 2 GATE PASS`, despite the OCI bootstrap script now never installing Node.js, breaking every OCI VPS deployment silently (Node-dependent MCP servers fail at runtime, not at provisioning time).
- **Evidence:** Reproduced live; "Tests After" (a separate, non-gating section) does check both files (`phase-02-nodesource-install-hardening.md:85-86`), which makes the gap in the actual gate more glaring — the stronger check exists in the plan but isn't wired into the pass/fail gate.
- **Suggested fix:** Duplicate both positive assertions for `vps-bootstrap-oci.sh` in the Regression Gate (it has no Caddy block to check, but it does need the `nodesource.gpg` assertion).

## Finding 5: "Apply Phase 1 before Phase 5" is unenforced prose, not a dependency — contradicted by the plan's own "no dependencies" declaration

- **Severity:** High
- **Location:** plan.md, section "Execution Strategy — `--parallel`" (`plan.md:50-57`) vs. `plan.md:69-71` ("Dependencies: None between phases") and phase frontmatter
- **Flaw:** `plan.md:52` states "no logical dependencies — none blocks another, all can run concurrently," then `plan.md:55` immediately contradicts this: "apply Phase 1's production.yaml edits before Phase 5's ... or serialize the two production.yaml edits under a single owner." This ordering requirement exists only as prose (repeated in `phase-05-cron-drift-reconciliation.md:52` and `:124`) — it is not encoded as a machine-checkable dependency anywhere. `grep -n 'blockedBy' phase-01-real-security-schema-rewrite.md phase-05-cron-drift-reconciliation.md` returns nothing; only `plan.md:9-10` has `blockedBy: []` / `blocks: []`, and both are empty.
- **Failure scenario:** Under a literal `--parallel` dispatch model (spawn all unblocked phases as concurrent subagents/processes, which is exactly what the plan recommends and what its "Suggested parallel grouping" table implies), nothing structurally prevents Phase 1 and Phase 5's subagents from both opening `templates/config/production.yaml` at the same instant. The plan's own conflict-prevention rule ("never a full-file Write, only scoped Edit") mitigates textual collision *if* both edits happen to target genuinely disjoint, already-existing text — but the explicit fallback instruction ("serialize the two production.yaml edits under a single owner") is itself an admission that the authors are not fully confident the scoped-Edit isolation is sufficient without ordering, yet no locking/ordering primitive is actually specified (no lockfile, no `blockedBy: [1]` on phase 5, no single-owner assignment named).
- **Evidence:** `plan.md:52` vs `plan.md:55`; frontmatter grep confirms no `blockedBy` field on either phase file.
- **Suggested fix:** Either declare `blockedBy: [1]` in phase-05's frontmatter (forcing sequential scheduling in any dependency-aware orchestrator) or explicitly instruct the orchestrator to route both `production.yaml` edits through a single subagent/turn rather than two concurrent ones. Prose alone will not survive a scheduler that only reads structured fields.

## Finding 6: Four byte-identical `trust: trusted` lines make the prescribed single scoped-Edit removal ambiguous

- **Severity:** Medium
- **Location:** Phase 1, section "Refactor" step 1 (`phase-01-real-security-schema-rewrite.md:120`), "Related Code Files" (`phase-01-real-security-schema-rewrite.md:96`)
- **Flaw:** The plan says "delete the `trust: trusted` line from each of the 4 `mcp_servers` entries" via a scoped `Edit`. All 4 occurrences are byte-identical: `grep -n 'trust: trusted' templates/config/production.yaml | cat -A` returns `139:    trust: trusted$`, `151:...`, `158:...`, `167:...` — same 4-space indent, no distinguishing suffix, under `github:`, `postgres:`, `cloudflare:`, `filesystem:` respectively.
- **Failure scenario:** A find/replace-style `Edit` tool that requires a unique `old_string` match (the class of tool this plan is designed to be executed by, per its own "never a full-file Write" rule) will reject a single-shot `old_string="    trust: trusted\n"` replacement as non-unique, since it matches 4 places. The plan doesn't call out using `replace_all: true` or supplying 4 distinct multi-line contexts (e.g. including each parent block's preceding unique line) to disambiguate — an executing agent will hit an avoidable tool error on the very first edit of the plan.
- **Evidence:** `grep -n 'trust: trusted' templates/config/production.yaml | cat -A` → 4 identical lines.
- **Suggested fix:** Explicitly instruct `replace_all: true` for this specific removal (safe here since the string is unique to trust lines), or provide the 4 distinct `old_string` snippets (each anchored to its parent server name) in the phase file.
