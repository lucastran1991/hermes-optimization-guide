## Finding 1: Phase 1's "ground truth" for `telegram-bot.yaml` is fabricated — the file has no `security:` block at all

The file has **no `security:` block anywhere** — `grep -n 'security:' templates/config/telegram-bot.yaml` returns zero matches, and the whole file is only 52 lines (`wc -l` = 52), not the ~70 lines the cited range implies.
- **Failure scenario:** Whoever executes Phase 1 literally cannot "replace" a block that isn't there. Two outcomes, both bad: (a) they notice the mismatch mid-execution and improvise a fix not specified anywhere in the plan (undocumented scope creep, no review coverage), or (b) they run the prescribed baseline grep, see `0` fictional-key hits (already zero, matching the "Tests After — expect 0" target *before any edit*), conclude the file needs no change, and skip it — silently leaving `telegram-bot.yaml` with **zero** operator guidance on `approvals:`/`security:` at all (pure implicit-default behavior, undocumented). Either way, Phase 1's own Success Criteria ("All 3 files contain only real hermes-agent keys in their security-related blocks") is unverifiable for this file because the premise is wrong. Root cause: the original scan report (`plans/reports/security-scan-260703-1005-full-repo-secrets-code-config-audit-report.md:46`) cites `telegram-bot.yaml:69` for an unrelated secrets-regex artifact, and the research report copied the file into HIGH #1's scope without re-reading current content — `git log --oneline -- templates/config/telegram-bot.yaml` shows the file was rewritten in commit `7771526` (2026-07-02, the day *before* the scan) to a flat `platforms.telegram` shape with no `security:` block.
- **Evidence:** `templates/config/telegram-bot.yaml` (52 lines total, `cat -n` confirms no `security:` key); `git show --stat 7771526` (rewrite commit, predates the 2026-07-03 scan); `phase-01-real-security-schema-rewrite.md:98,122` (cites a block that doesn't exist).
- **Suggested fix:** Re-verify all 3 target files' current content before finalizing Phase 1 (not just at research time — re-check at execution time, since templates were mid-rewrite the day before the scan). For `telegram-bot.yaml`, either drop it from Phase 1's scope entirely (nothing fictional to remove) or explicitly scope a new task: *add* a minimal `approvals:`/`security:` block with real keys, since none exists today and the file currently ships on 100% implicit defaults.

## Finding 2: Phase 4's "commit SHA" is the git tag object SHA, not the commit it points to

- **Severity:** Critical
- **Location:** Phase 4, section "Architecture" (line 19: *"Pin to the commit SHA `499c1e7f3637c131334fa8e937c45144f79d72d2`"*)
- **Flaw:** `git ls-remote https://github.com/gaurav-nelson/github-action-markdown-link-check.git 'refs/tags/v1*'` returns two lines for an annotated tag — the tag object's own SHA and, marked `^{}`, the commit it dereferences to:
  ```
  499c1e7f3637c131334fa8e937c45144f79d72d2  refs/tags/v1
  5c5dfc0ac2e225883c0e5f03a85311ec2830d368  refs/tags/v1^{}
  ```
  The plan (and research report line 103) transcribed the first line and called it "the commit SHA." Confirmed via GitHub API: `GET /repos/gaurav-nelson/github-action-markdown-link-check/git/tags/499c1e7f...` returns `200` with `"object": {"sha": "5c5dfc0ac2e225883c0e5f03a85311ec2830d368", "type": "commit"}`; `GET /repos/.../git/commits/499c1e7f...` returns `404 Not Found` — `499c1e7f...` is unambiguously a tag object, not a commit.
- **Failure scenario:** `codeload.github.com/.../tar.gz/499c1e7f...` and `.../tar.gz/5c5dfc0a...` both currently return `200` with an identical ETag (GitHub's tarball endpoint dereferences transparently today), so the CI job will likely keep running. But the pin fails its actual purpose: (1) an annotated tag object is only reachable via the `v1` ref — if the maintainer deletes/moves `v1`, the orphaned tag object becomes eligible for Git garbage collection, and a future `uses:` resolution can fail with "reference not found" for reasons unrelated to any real compromise; (2) tooling that audits/rotates SHA-pinned actions (Dependabot's `github-actions` SHA-update flow, OpenSSF Scorecard, `zizmor`, `pin-github-action`) resolves via the commits API (`/git/commits/{sha}`), which 404s for this SHA — such tooling will not recognize this pin as pointing at `v1` and may silently stop proposing updates for this action, undermining the audit trail the whole phase exists to create.
- **Evidence:** `git ls-remote` output above; GitHub API responses (`/git/tags/499c1e7f...` → 200, `/git/commits/499c1e7f...` → 404); `phase-04-ci-action-sha-pin.md:19,42,74`.
- **Suggested fix:** Pin to `5c5dfc0ac2e225883c0e5f03a85311ec2830d368 # v1` (the peeled/`^{}` commit), not `499c1e7f...`. Update the "Tests After" `git ls-remote` step to explicitly instruct reading the `^{}`-suffixed line when the tag is annotated.

## Finding 3: `security-hardened.yaml`'s entire `profile`/`profiles:` mechanism — its stated core feature — is fictional; Phase 1 only patches one nested key inside it

- **Severity:** Critical
- **Location:** Phase 1, "Related Code Files" step 2 (scope limited to `profiles.quarantine.security.*` ~24-29 and top-level `security:` ~60-95)
- **Flaw:** `security-hardened.yaml`'s own header (lines 1-17) markets "Quarantine mode as default" as the file's headline protection, implemented via `profile: quarantine` (line 17) selecting into a `profiles: {quarantine, trusted}` dict (lines 19-31) with per-profile `models`, `tools_allowlist`, `memory: {write,read}`. `grep -n '"profiles"\|"profile"' hermes_cli/config.py` returns **zero hits** in DEFAULT_CONFIG — this top-level selector/dict structure is not read by the config loader at all, not just its nested `security:` sub-key (which is all Phase 1 touches). Additionally, the file's own comment "explicit `/trusted` to leave" implies a slash command that also doesn't exist: `grep -rn "'/trusted'"` across all `.py` in hermes-agent returns zero hits.
- **Failure scenario:** Post-Phase-1, an operator running this "security-hardened" template for a public/webhook-facing bot reads `profile: quarantine` at the top and reasonably believes untrusted conversations get the cheap-model/no-memory-write/restricted-tools treatment described in lines 1-31. In reality every conversation runs under whatever the single top-level `approvals:`/`security:` block enforces — there is no runtime profile-switching at all. Phase 1's fix (a pointer-comment inside `profiles.quarantine.security`) leaves the surrounding `profile:`/`profiles:` scaffold, the `/trusted` reference, and the per-profile `models`/`tools_allowlist`/`memory` keys completely untouched and still visually asserting the mechanism is active. This is a larger capability gap than any of the 3 already-documented capability losses, yet it's folded into a Med/Low risk row ("`profiles:` block still fictional after rewrite") rather than surfaced as its own item.
- **Evidence:** `templates/config/security-hardened.yaml:1-31`; zero hits for `"profiles"`/`"profile"` as a config-schema key in `hermes_cli/config.py` (the only `profiles[...]` hit in the whole tree is `hermes_cli/main.py:8740`, an unrelated PID-tracking dict); zero hits for `/trusted` as a slash command.
- **Suggested fix:** Add a 4th, more prominent capability-loss disclosure specifically about the file's core "quarantine vs trusted" premise (not folded into the generic risk table), or rewrite the header + `profile:`/`profiles:` scaffold to either point at the real (unrelated) directory-based `hermes -p <name>` profile feature or remove the scaffold outright. Surface as its own validate-workflow question — this is bigger than unresolved item 1.

## Finding 4: hermes-agent's real webhook adapter already enforces rate-limiting and `max_body_bytes` — Phase 1's comment and Phase 3's rewritten comment both introduce a new false claim while fixing the old one

- **Severity:** High
- **Location:** Phase 1, "Implementation Steps" step 1 (*"body-size is capped in Caddy"*); Phase 3, "Architecture" recommended comment (*"no request-RATE limit is enforced — stock Caddy has none"*)
- **Flaw:** Both phases treat the Caddy layer (`request_body { max_size 1MB }`) as the only enforcement point for the webhook vhost and conclude there's no config-level rate limit or body cap on the hermes-agent side ("No `max_body_bytes`/`ttl_seconds` config exists anywhere" — research report line 65). This is false: `gateway/platforms/webhook.py:145` sets `self._rate_limit: int = int(config.extra.get("rate_limit", 30))  # per minute`, enforced via `_record_rate_limit_hit` (`:316-339`, checked at `:511`); `:148-149` sets `self._max_body_bytes: int = int(config.extra.get("max_body_bytes", 1_048_576))`, enforced at `:474`. Both are real, user-configurable keys under `platforms.webhook.extra.*`.
- **Failure scenario:** Phase 3 ships a *new* factually incorrect claim ("no request-RATE limit is enforced") into a comment whose whole purpose is fixing a factually incorrect claim — worse than the bug it replaces, since it now flatly contradicts the running system's actual behavior. Separately, `templates/config/production.yaml` has no `platforms.webhook` section at all (`grep -n 'platforms:' ` shows only telegram/discord/slack/google_chat/teams/line/simplex/email — no `webhook:` entry), so the real `rate_limit`/`max_body_bytes` knobs aren't even exposed to the operator in the template Phase 1 is rewriting for accuracy — the phase fixes the fictional `security.webhook.*` keys but doesn't add the real `platforms.webhook.extra.*` equivalent it just learned about.
- **Evidence:** `gateway/platforms/webhook.py:145,148-149,316-339,474,511`; `templates/config/production.yaml` (`platforms:` block, lines 61-118, no `webhook:` key); `phase-01-real-security-schema-rewrite.md:120`; `phase-03-caddyfile-rate-limit-claim-fix.md:19,45`.
- **Suggested fix:** Reword Phase 3's comment to state rate limiting is enforced by hermes-agent's webhook adapter (default 30 req/min/route via `platforms.webhook.extra.rate_limit`), not Caddy; Caddy only enforces the 1MB body cap pre-auth. Update Phase 1's comment similarly and consider adding a real `platforms.webhook.extra: { rate_limit, max_body_bytes }` example to `production.yaml` since the template currently doesn't configure the webhook platform at all despite the Caddyfile routing to it.

## Finding 5: Phase 1 explicitly leaves `trust_label`, `allow_sampling`, `tools_allowlist` untouched — confirmed equally fictional, and far more widespread than acknowledged

- **Severity:** High
- **Location:** Phase 1, "Key Insights" line 31 (*"not flagged by research → out of scope, leave untouched"*)
- **Flaw:** `grep -rn 'allow_sampling\|tools_allowlist' --include='*.py' .` in hermes-agent returns **zero** hits. The real per-MCP-server sampling config is a nested dict (`tools/mcp_tool.py:44` docstring: `sampling: { enabled: true, model, max_tokens_cap, timeout, max_rpm, allowed_models, max_tool_rounds, log_level }`), not a flat `allow_sampling: false`. There is no real per-server tool-restriction mechanism at all (zero hits for `allowed_tools`/`tool_filter`/`allowlist` in `tools/mcp_tool.py`). `trust_label` likewise has zero hits as a `platforms.*` config key anywhere in `.py` source (its only real use is an unrelated skill-marketplace display field in `hermes_cli/skills_hub.py`) — yet `production.yaml`'s `platforms:` block sets `trust_label:` on **8** separate platform entries (telegram, discord, slack, google_chat, teams, line, simplex, email), and `mcp_servers.github/postgres/cloudflare` set `allow_sampling: false` one line below the exact `trust: trusted` line Phase 1 is deleting in the same stanza.
- **Failure scenario:** Post-Phase-1, `production.yaml`'s `mcp_servers.github` entry reads `tools_allowlist: [get_pull_request, list_pull_requests, ..., create_pull_request_review, create_pull_request]` — an operator reasonably believes this restricts the GitHub MCP server to only these tool calls (a genuine access-control expectation on a token-bearing, write-capable server). It does nothing. Phase 1's stated architecture goal — "what the file *says* equals what the agent *enforces*" — is violated by design, in the same `mcp_servers` block the phase is editing, for the same class of bug (HIGH #1) the phase exists to fix.
- **Evidence:** `templates/config/production.yaml:61-118` (8x `trust_label:`), `:140-141,152,159` (`allow_sampling`/`tools_allowlist` beside deleted `trust:` lines); `tools/mcp_tool.py:14-49` (real nested `sampling:` schema, no tools-restriction key); `hermes_cli/skills_hub.py` (only real `trust_label` usage, unrelated subsystem).
- **Suggested fix:** Re-scope Phase 1 to include these three keys with the same "confirmed fictional" verdict derived here — either delete them with a capability-loss comment (matching the `trust:` treatment) or map `allow_sampling: false` to the real nested `sampling: { enabled: false }` key, which does have a real equivalent (unlike the other two).

## Finding 8: Phase 5's reconciliation policy propagates duplicate cron trigger times without flagging simultaneous-trigger contention

- **Severity:** Medium
- **Location:** Phase 5, "Requirements" (*"weekly-cost-report→9am Mon, weekly-dep-audit→9am Mon"*, *"weekly-mcp-audit→10am Mon"*)
- **Flaw:** `production.yaml`'s cron block (the designated source of truth) already has `weekly-dep-audit` and `weekly-cost-report` both scheduled `"0 9 * * 1"` (Monday 9am) — confirmed via direct read of `templates/config/production.yaml:290-293`. Phase 5's reconciliation policy aligns `production-crons.yaml`'s matching jobs to these same times. After reconciliation, `production-crons.yaml` will have **two pairs** of jobs firing at the identical cron expression: `weekly-cost-report`/`weekly-dep-audit` both at `0 9 * * 1`, and `weekly-mcp-audit` (aligned to 10am per the plan) colliding with the pre-existing, kept-as-unique `weekly-bypass-audit`, also `0 10 * * 1` (`templates/cron/production-crons.yaml:17-19`, unchanged by the plan).
- **Failure scenario:** Two agent-loop cron jobs firing at the exact same wall-clock minute means two concurrent LLM sessions launch simultaneously (cost-report + dep-audit; mcp-audit + bypass-audit) — a cost/rate-limit spike the operator didn't design for, and if `approvals.cron_mode: deny` handling or shared state (e.g. `~/.hermes/shell-hooks-allowlist.json`, cost-tracking counters) isn't safe under concurrent cron execution, this is a genuine shared-mutable-state race the plan doesn't address. Phase 5's own Risk Assessment table doesn't mention same-time collisions at all — only channel/schedule drift.
- **Evidence:** `templates/config/production.yaml:290-293` (weekly-dep-audit and weekly-cost-report both `"0 9 * * 1"`); `templates/cron/production-crons.yaml:17-19` (weekly-bypass-audit `"0 10 * * 1"`, a "keep as unique" job per Phase 5's own Requirements); `phase-05-cron-drift-reconciliation.md:33` (alignment instructions that create the second collision).
- **Suggested fix:** Either stagger the aligned times by a few minutes when reconciling, or explicitly verify (and note in Security Considerations) that hermes-agent's cron scheduler safely parallelizes same-time triggers before accepting the collision as a non-issue.
