## Finding 1: Phase 4 pins to the annotated-tag object SHA, not the commit SHA
- **Severity:** Critical
- **Location:** Phase 4, section "Architecture" / "Regression Gate"
- **Flaw:** `git ls-remote` on an annotated tag returns two lines — the tag object's own SHA, and (with `^{}` suffix) the commit it points to. The plan transcribed the tag-object SHA and calls it "the commit SHA."
- **Failure scenario:** `git ls-remote https://github.com/gaurav-nelson/github-action-markdown-link-check.git 'refs/tags/v1*'` returns:
  ```
  499c1e7f3637c131334fa8e937c45144f79d72d2  refs/tags/v1
  5c5dfc0ac2e225883c0e5f03a85311ec2830d368  refs/tags/v1^{}
  ```
  Confirmed via GitHub API (`GET /repos/gaurav-nelson/github-action-markdown-link-check/git/tags/499c1e7f...`): `"object": {"sha": "5c5dfc0ac2e225883c0e5f03a85311ec2830d368", "type": "commit"}`. `499c1e7f...` is the **tag object**, `5c5dfc0a...` is the **commit**. Pinning `uses:` to a tag-object SHA is the exact anti-pattern SHA-pinning exists to avoid: the tag object is only reachable via the `v1` ref; if the maintainer ever deletes/re-creates the `v1` tag, the tag object can become unreferenced and eligible for GC, silently breaking the workflow (`unable to resolve action` at a random future date) — whereas the underlying commit is far more likely to stay reachable independent of the tag. Some SHA-verification tooling (dependabot, `pin-github-action`, OpenSSF scorecard) also expects a commit SHA and will flag/rewrite a tag-object pin.
- **Evidence:** `phase-04-ci-action-sha-pin.md:19,42`: `"Pin to the commit SHA 499c1e7f3637c131334fa8e937c45144f79d72d2 (research-verified via git ls-remote 2026-07-03, current v1 target)"`. `research/researcher-real-hermes-schema-and-fix-verification-report.md:103` makes the same misattribution.
- **Suggested fix:** Use `5c5dfc0ac2e225883c0e5f03a85311ec2830d368 # v1` (the peeled/`^{}` SHA), not `499c1e7f...`. Regression-gate should assert against the peeled SHA specifically, and the "Tests After" `git ls-remote` step should document that annotated tags require reading the `^{}` line.

## Finding 2: Phase 3's rewritten comment introduces a new, verifiably false claim

The full request path (Caddy → hermes webhook listener) **is** rate-limited today — just not by Caddy. Shipping the plan's proposed comment replaces a misattributed-but-directionally-true claim with a flatly false one ("no request-RATE limit is enforced"), which is worse than the original bug for a docs-accuracy fix.
- **Evidence:** `templates/caddy/Caddyfile:7` (`hooks.yourdomain.com -> Hermes webhook listener (127.0.0.1:8766)`); `gateway/platforms/webhook.py:145,316-339,511`; `website/docs/user-guide/messaging/webhooks.md:398-410` (rate-limiting section, `rate_limit: 60` example).

## Finding 5: The `profiles:`/`profile:` mechanism itself — not just its nested `security:` block — appears to be fictional; Phase 1 only fixes the inner layer
