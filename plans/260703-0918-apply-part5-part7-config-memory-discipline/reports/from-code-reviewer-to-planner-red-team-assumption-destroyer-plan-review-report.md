# Red Team Review — Assumption Destroyer (Fact Checker + Contract Verifier)

Plan: `plans/260703-0918-apply-part5-part7-config-memory-discipline/`
Reviewer role: hostile, independent re-verification (planner's own Red Team section not taken at face value).

## Finding 1: `curator:` config block schema is invented wholesale — guide never shows curator as YAML-configurable

- **Severity:** Critical
- **Location:** Phase 1, section "Implementation Steps" step 2 (the `curator:` block) and "Key Insights" ("Curator is a system feature... Config-only is the correct application")
- **Flaw:** Every occurrence of Curator in the source guide describes it as a **CLI-only** subsystem: `hermes curator run --dry-run`, `hermes curator run`, `hermes curator enable`, `hermes curator status` (`part5-creating-skills.md:123-132`, repeated verbatim at `part22-latest-power-moves.md:9-14`). Nowhere does the guide show or imply a `curator:` top-level YAML key, nor keys named `enabled`, `schedule`, `dry_run_schedule`, `auto_archive`, `pinned_skills`, or `notify` under it. `grep -rn "curator:" part*.md` returns zero matches — confirmed independently.
- **Failure scenario:** An end user copies `production.yaml` into `~/.hermes/`. Hermes's real config loader (which this repo cannot exercise — "no live Hermes runtime") either (a) silently ignores the unknown `curator:` block (per the plan's own Risk Assessment row "Curator engine ignores unknown keys, block is inert"), giving the user false confidence that weekly hygiene is scheduled when nothing is, or (b) errors on an unrecognized top-level key if Hermes validates config strictly — neither behavior is verifiable from this repo, and the plan ships it as fact regardless.
- **Evidence:** `part5-creating-skills.md:121-133` shows only bash CLI invocations. `part22-latest-power-moves.md:9-14` repeats the same three CLI commands, no YAML. `grep -n "curator" templates/config/*.yaml` (pre-change) returns nothing — no existing precedent for a curator config schema in this repo either.
- **Suggested fix:** Either cut the `curator:` config block entirely and instead document `hermes curator enable` as an operator action in a runbook/README, or explicitly label the block `[SPECULATIVE — schema not confirmed against Hermes source, verify before relying on it]`. The plan's own "no invented scope" principle (which caught RT-1's fabricated weights) should have caught this larger fabrication — a whole config surface, not just tuning values.

## Finding 2: `memory_audit:` assumes `/journey` is schedulable and can emit an automated report — the guide says the opposite

- **Severity:** Critical
- **Location:** Phase 1, section "Implementation Steps" step 3 (the `memory_audit:` block) and Architecture diagram (`memory_audit.schedule → /journey review pass → REPORT`)
- **Flaw:** `part7-memory-system.md:120,123` and `part26-moa-verification.md:93,98,101,168` describe `/journey` exclusively as an **interactive, human-driven** command: "a timeline... edit or delete any entry in place," "`/journey` works in the CLI and TUI; the desktop app adds a memory graph," "Do a `/journey` pruning pass monthly," "Run `/journey` once — prune anything wrong." There is no mention anywhere of `/journey` running headlessly on a cron trigger, producing a machine-readable "report," or emitting output to a notify channel. This is categorically different from Curator, which at least has an explicit non-interactive `run`/`status` CLI mode the guide documents as automatable.
- **Failure scenario:** The `memory_audit.schedule: "0 2 1 * *"` fires monthly expecting `/journey` to run unattended and push a stale/suspicious-entry summary to `telegram_dm`. If `/journey` has no headless mode (as every citation implies), the scheduled job either does nothing, errors silently, or (worst case) an operator who read the config comment ("Monthly memory-hygiene REPORT... does NOT delete") believes hygiene review is happening automatically every month and stops doing the manual `/journey` pass the guide actually recommends — the opposite of the intended safety effect.
- **Evidence:** `part7-memory-system.md:120-123` ("open `/journey`", never "schedule /journey"); `part26-moa-verification.md:98` ("`/journey` works in the CLI and TUI"). `suspicious_patterns` regex matching and `stale_threshold_days` filtering (Phase 1 lines 126-131) have zero textual basis anywhere in `part7-memory-system.md` — the anti-pattern table (`part7:125-134`) is written as human/agent judgment guidance ("Save task progress to memory" → don't), not a technical spec for an automated scanner.
- **Suggested fix:** Reframe `memory_audit:` as a `cron:` entry that sends a monthly **reminder** to run `/journey` manually (matches how `part7:123` and `part26:168` actually describe the cadence), rather than a block implying Hermes itself scans and reports memory content unattended.

## Finding 3: `pinned_skills` protects a category Curator already excludes by design

- **Severity:** High
- **Location:** Phase 1, Implementation Steps step 2, `pinned_skills` field and its "Traces to" citation `part5:140,145`
- **Flaw:** `part5-creating-skills.md:141` states Curator "Focuses on agent-created skills first, not bundled/vendor skills." The four names in `pinned_skills` — `coding-agent-delegate`, `nightly-backup`, `audit-mcp`, `rotate-secrets` — are this repo's own shipped example skills under `skills/dev/`, `skills/ops/`, `skills/security/` (confirmed: `skills/dev/coding-agent-delegate`, `skills/ops/nightly-backup`, `skills/security/audit-mcp`, `skills/security/rotate-secrets` all exist as bundled catalog dirs, per `skills/README.md:41` and directory listing). These are exactly the "bundled/vendor skills" Curator already skips per `part5:141` — pinning them is redundant, and citing `part5:140,145` ("pin your production runbooks") without acknowledging `part5:141` right next to it is a selective read of the same section.
- **Failure scenario:** An operator reads `pinned_skills` and believes these four are at risk of auto-archival without the pin (false threat model), while the actual risk category (real agent-created skills accumulating in the user's live `~/.hermes/skills/` after months of "save that as a skill") goes completely unaddressed by this list, since none of those skills exist yet in a template repo.
- **Evidence:** `part5-creating-skills.md:135-141` (bullets listing all 5 Curator behaviors, including line 141 exclusion clause) vs. Phase 1's citation of only `:140,145`.
- **Suggested fix:** Either drop `pinned_skills` (nothing in this repo needs it, since none of the shipped skills are in Curator's scope), or add an explicit comment acknowledging these are illustrative examples, not an exhaustive/necessary list, and that real pin candidates are whatever agent-created skills a live install accumulates.

## Finding 4: No schema validation exists — a typo'd key in either new block ships silently to every downstream user

- **Severity:** High
- **Location:** Phase 1, Success Criteria ("YAML parses clean") and plan.md's claim "No CI change (`validate_skills.py` only validates `skills/**`, not config)"
- **Flaw:** Confirmed `validate_skills.py:88` does `root.rglob("SKILL.md")` — it never touches `templates/config/*.yaml`, so the plan's claim is technically true. But the plan treats "YAML parses" (syntactic validity) as sufficient success criteria, ignoring that CI *does* run a yaml-lint job against `templates/` (`.github/workflows/ci.yml:24-32`, confirmed: `yamllint -c .github/yamllint.yml templates/ benchmarks/ skills/`). That job has `comments`, `comments-indentation`, `colons`, `commas`, `indentation`, and `line-length` all **disabled** (`.github/yamllint.yml:8-14`) specifically to allow template flexibility. So a typo like `enabld: true`, `dry_run_schedul:`, or a wrong indent level under `curator:` would still pass both `yaml.safe_load` and yamllint — nothing in this repo's toolchain can catch a semantically wrong key name, and (per Finding 1/2) there is no confirmed real schema to check it against even if a validator existed.
- **Failure scenario:** A future edit typos `auto_archive` as `autoarchive` during a later refactor. `git diff` "additive only" check (Phase 1 step 5) doesn't catch semantic typos. CI stays green. The field silently does nothing on every end-user install that copies the template, and nobody notices until someone manually diffs against the (nonexistent, per Finding 1) real Hermes schema.
- **Evidence:** `.github/scripts/validate_skills.py:88`; `.github/workflows/ci.yml:24-32`; `.github/yamllint.yml:7-17`.
- **Suggested fix:** State explicitly in Phase 1 Risk Assessment that no CI mechanism validates config *semantics*, only syntax — this is a known, accepted gap, not an oversight, so a future reader doesn't assume yamllint is a safety net it isn't.

## Finding 5: Phase 2's insertion-point instructions ignore an intervening `## Triggering` section

- **Severity:** Medium
- **Location:** Phase 2, "Implementation Steps" step 1 ("Locate the end of `## Procedure` (line 75, step 5) and the start of `## Tips` (line 88)") and Requirements ("Placed after `## Procedure` and before `## Tips`")
- **Flaw:** Between `## Procedure` (ends line 75) and `## Tips` (starts line 88) sits a third, separately-headed section: `## Triggering 15 min before each meeting` (`skills/dev/meeting-prep/SKILL.md:77-86`), containing a `cron:` YAML example. The plan's framing ("after Procedure, before Tips") is literally true but conceptually wrong — it never acknowledges this third section exists, so "insert between them" is ambiguous: immediately after line 75 (splitting `## Procedure` from the `## Triggering` section that documents how the procedure actually fires) vs. immediately before line 88 (after `## Triggering`, which is what the line numbers in the plan actually resolve to).
- **Failure scenario:** An implementer following the prose literally ("after Procedure") could insert the new `## Memory discipline` section directly at line 76, before `## Triggering 15 min before each meeting` — separating the 5-step procedure from its own triggering-mechanism documentation, which currently reads as one continuous flow (procedure → how it's invoked on a schedule → tips). This produces a worse document structure than either "before Procedure ends" or "after Triggering ends" cleanly.
- **Evidence:** `skills/dev/meeting-prep/SKILL.md:75` (step 5, end of Procedure), `:77` (`## Triggering 15 min before each meeting` header), `:88` (`## Tips` header) — confirmed via direct read, three distinct H2-delimited sections exist where the plan describes only two.
- **Suggested fix:** Explicitly say "insert immediately after the `## Triggering...` section closes (after the cron yaml block, line 86), immediately before `## Tips`" — remove the "after Procedure" framing that omits the section in between.

## Finding 6: `memory_audit` report payload risk not analyzed — same channel, more sensitive content class

- **Severity:** Medium
- **Location:** Phase 1, "Security Considerations" ("notify: telegram_dm reuses the already-configured approval/cron channel; introduces no new outbound destination") and Red Team RT-2 disposition ("Reject — no new egress surface")
- **Flaw:** RT-2's rejection reasons only about *channel* novelty (telegram_dm already used), never about *payload sensitivity*. The existing `telegram_dm` consumers (`cron:` entries at `production.yaml:290-294`: nightly-backup confirmation, dep-audit findings, cost report, MCP audit, secret-rotation confirmation) are operational metadata. A `memory_audit` report — even assuming Finding 2's automation gap is somehow solved — would summarize *contents* of persistent memory, which per Phase 2's own Security Considerations "may include sensitive attendee/deal context." Flagging "suspicious" memory entries for a Telegram DM report is a materially different data-exposure class than a backup-completion ping, and the plan's security analysis never distinguishes the two.
- **Failure scenario:** A `memory_audit` report includes an excerpt of a flagged stale/suspicious memory entry containing a partial deal term or attendee detail (to let the operator identify which entry to review) and pushes it to a Telegram DM — a channel with weaker at-rest/transport guarantees than the memory store itself, and outside the `security.secrets.redaction_patterns` scope (`production.yaml:266-272`), which targets API-key-shaped secrets, not free-text PII.
- **Evidence:** `production.yaml:266-272` (`redaction_patterns` list is all credential regexes: `sk-`, `xoxb-`, `ghp_`, `github_pat_`, `AKIA` — none target personal/relationship data); Phase 2 Security Considerations explicitly names "sensitive attendee/deal context" as a risk category for memory content.
- **Suggested fix:** If `memory_audit` reporting is kept (pending Finding 2's resolution), specify that reports include entry IDs/timestamps only, never content excerpts, or note this as an explicit open risk rather than folding it into "no new egress surface = no new risk."
