## Finding 1: `memory_audit:` invents a scheduling/automation mechanism the guide never describes

- **Severity:** Critical
- **Location:** Phase 1, section "Insert the `memory_audit:` block" (fields `schedule`, `suspicious_patterns`, `enabled`)
- **Flaw:** The plan models `memory_audit:` as a cron-schedulable, config-driven job (`schedule: "0 2 1 * *"`, regex `suspicious_patterns`, `notify: telegram_dm` report) analogous to `curator:`. But `curator:` has textual justification for schedulability (`hermes curator enable` / `hermes curator status` — a documented CLI subcommand pair, `part5-creating-skills.md:128-132`). `/journey` has **no such equivalent anywhere in the guide**. Every mention of `/journey` across `part7-memory-system.md`, `part22-latest-power-moves.md`, and `part26-moa-verification.md` describes it as an interactive, human-run command: "Do a `/journey` pruning pass monthly" (`part7-memory-system.md:123`), "open `/journey`, prune wrong or stale memories" (`part26-moa-verification.md:101`), "Run `/journey` once — prune anything wrong before it compounds" (`part26-moa-verification.md:168`). Nowhere does the guide mention a `hermes journey enable/schedule/status` CLI, a cron-triggerable audit pass, or a regex-based "suspicious pattern" flagging capability.
- **Failure scenario:** An operator ships this config expecting a monthly automated report to land in their Telegram DM (per the block's own comment: "Monthly memory-hygiene REPORT... does NOT delete"). If the underlying Hermes engine has no mechanism to schedule `/journey` (because none is documented — matching the plan's own Risk row "Curator engine ignores unknown keys, block is inert"), the block silently does nothing. The operator believes memory hygiene is being audited monthly and stops doing the guide's actual mandated action ("Do a `/journey` pruning pass monthly" — a manual human action) because they think config now automates it. This is worse than the curator case because curator's block at least maps 1:1 to a real CLI feature; `memory_audit:` maps to nothing.
- **Evidence:** `part7-memory-system.md:120-123` ("`/journey` — a timeline... edit or delete any entry in place... Do a `/journey` pruning pass monthly"); `part26-moa-verification.md:86-101` (no CLI schedule verb, only `/learn <anything>` and `/journey` as slash commands); grep of all three guide files for "schedule|cron|automat" near journey returns zero hits associating `/journey` with automation. The plan's own Red Team (RT-1, `plan.md:92`) caught the *identical* fabrication class for `curator`'s scoring weights but never applied the same "does the CLI/mechanism actually exist" check to `memory_audit`'s core premise — the plan is inconsistent about which invented capabilities get cut.
- **Suggested fix:** Cut `memory_audit:` down to documenting the *guide's actual* mechanism — e.g. a `cron:` entry that runs a `/journey`-adjacent report skill if one exists, or, if no such automation exists, replace the block with a comment-only reminder (matching how the plan already handled the scoring-weight cut) rather than shipping a schedule/regex config that has no execution path.

## Finding 2: `security-hardened.yaml` exclusion is unreasoned and inconsistent with the plan's own inclusion logic

- **Severity:** High
- **Location:** plan.md, "Overview" (line 46-47); Phase 1 "Overview" (line 26)
- **Flaw:** The plan's sole justification for skipping `security-hardened.yaml` is: "No `security-hardened.yaml` change (admin config, not conversation capability)" (`plan.md:46-47`). But this is the *same* justification used to argue the blocks are safe to add to `production.yaml` without approval-gating ("admin-only config... not new tool call surfaces", Phase 1 Security Considerations, `phase-01...md:168`). The rationale is self-contradictory: "admin config, not conversation capability" is presented as a reason to both include (production.yaml, safe/no-gate-needed) and exclude (security-hardened.yaml) the identical blocks. Meanwhile `security-hardened.yaml` is the one profile — besides production — that actually **has** a `cron:` block (`templates/config/security-hardened.yaml:110-119`) with its own dedicated security-audit cadence (`weekly-mcp-audit`, `weekly-bypass-audit`, `monthly-rotate`, `daily-log-sweep`), i.e. the scheduling infrastructure curator/memory_audit piggyback on already exists there. The other 3 excluded templates (`minimum.yaml`, `cost-optimized.yaml`, `telegram-bot.yaml`) have **no** `cron:` block at all (`grep -n "cron:" templates/config/*.yaml` returns only `production.yaml:289` and `security-hardened.yaml:110`), so their exclusion is at least structurally consistent — `security-hardened.yaml`'s is not.
- **Failure scenario:** A user who adopts the security-hardened profile (untrusted-input-facing bots, per its own header comment) never gets curator skill hygiene or memory-audit guidance in their template, even though this profile is exactly the "paranoid" tier most likely to accumulate agent-created skills/memory entries under adversarial conditions worth auditing. They have to manually port the blocks from `production.yaml`, and the plan gives zero guidance that this is a manual step they should take — it frames the omission as a deliberate scope decision without engaging the actual reason `security-hardened.yaml` is different (it already has scheduling infra + audit-focused posture, unlike the other 3 excluded templates).
- **Evidence:** `templates/config/security-hardened.yaml:1-13` (header: "Periodic security crons" is a stated design goal of this profile); `:23` (`memory: { write: false, read: true }` — read access to a shared memory store persists even here, so stale/suspicious entries injected from other profiles' writes still affect this profile); `:110-119` (existing `cron:` block).
- **Suggested fix:** Either (a) explicitly evaluate and reject security-hardened.yaml with a reason tied to its actual properties (e.g. "memory: write:false means memory_audit has nothing new to write, and existing daily-log-sweep/weekly-bypass-audit already cover its housekeeping cadence"), or (b) add a scoped `curator:`/`memory_audit:` variant there too. Current text is a scope gap dressed as a decision.

## Finding 3: `memory_audit`'s Telegram report has no redaction path — flagged "suspicious" entries could leak secrets/PII unredacted

- **Severity:** High
- **Location:** Phase 1, "Exact block to insert" for `memory_audit:` (`notify: telegram_dm`); Security Considerations section
- **Flaw:** The block's purpose is to "flag stale and anti-pattern entries for human review" via a Telegram DM report. To be actionable for the operator to judge "is this memory entry wrong," the report almost certainly must include the *content* of the flagged memory entries (not just IDs) — otherwise the operator can't decide anything from the DM. The repo already has an explicit redaction mechanism for memory: `security.secrets.redaction_patterns` + `memory_write_redaction: true` (`templates/config/production.yaml:265-272`), but that only fires on **write**. Nothing in the `memory_audit:` design or its Security Considerations section (`phase-01...md:166-170`) addresses whether the audit *report* content passes through the same redaction patterns before being sent over Telegram.
- **Failure scenario:** A memory entry containing a secret slipped past write-time redaction (redaction patterns are regex-based and inherently incomplete — e.g. a raw password, an internal hostname, or PII like a phone number that doesn't match `sk-`/`ghp_`/`AKIA` patterns) sits flagged as "suspicious" (matches `TODO|task|progress` or `temporary|temp|WIP`... note these patterns wouldn't even catch a secret, they catch task-state language, so *any* secret that happens to also read as stale/suspicious for other reasons gets surfaced). The audit report forwards the raw entry text to `telegram_dm` for operator review — the exact same "no new outbound destination" framing in Security Considerations (`phase-01...md:170`) is used to wave off scrutiny of *what* now flows through that destination, when the actual risk is report *content*, not destination novelty.
- **Evidence:** `templates/config/production.yaml:265-272` (redaction only scoped to `secrets:` config + `memory_write_redaction`, no `memory_read_redaction` or audit-report analog exists); Phase 1 Security Considerations only discusses whether `curator.auto_archive` needs an approval gate, never whether `memory_audit`'s outbound report content needs redaction.
- **Suggested fix:** Explicitly state whether the audit report is redacted (reusing `security.secrets.redaction_patterns`) before send, or restrict the report to metadata (entry ID, age, matched-pattern name) without echoing raw entry content.

## Finding 4: `pinned_skills` protects skills curator's own design already deprioritizes for archival — the mitigation is inert

- **Severity:** Medium
- **Location:** Phase 1, `curator.pinned_skills` field (`coding-agent-delegate`, `nightly-backup`, `audit-mcp`, `rotate-secrets`)
- **Flaw:** `part5-creating-skills.md:141` states Curator "Focuses on agent-created skills first, not bundled/vendor skills." All four pinned entries are this repo's own **bundled/vendor** skills (verified: `skills/dev/coding-agent-delegate`, `skills/ops/nightly-backup`, `skills/security/audit-mcp`, `skills/security/rotate-secrets` all exist under `skills/`, shipped in this repo, not agent-created at runtime). Per the guide's own description, curator wasn't going to archive them anyway — pinning provides protection against a risk class (`part5:141`) the guide says curator doesn't primarily target.
- **Failure scenario:** Reviewer/operator reads `pinned_skills` and concludes "our critical runbooks are protected from curator." The actual risk curator addresses — agent-created skills accumulating in an end-user's live `~/.hermes/skills/` after months of "save that as a skill" (`part5:119`) — is exactly what `pinned_skills` can't cover, because those skills don't exist yet at template-authoring time and aren't named in this shipped config. The list gives false confidence about coverage while providing near-zero real protection.
- **Evidence:** `part5-creating-skills.md:141` ("Focuses on agent-created skills first, not bundled/vendor skills"); `find skills -iname "*<name>*"` confirms all four are checked-into, vendor-shipped skill directories, not agent-created artifacts.
- **Suggested fix:** Either drop `pinned_skills` from the template (nothing to pin at ship time) or reframe the comment to clarify it's a *documentation example* of the field's shape for users to fill in with their own agent-created runbooks later, not a functioning protection for the 4 named skills.

## Finding 5: Phase 2 placement instructions ignore an entire intervening section, contradicting the "adjacent" framing

- **Severity:** Medium
- **Location:** Phase 2, "Implementation Steps" step 1: "Locate the end of `## Procedure` (line 75, step 5) and the start of `## Tips` (line 88). Insert this section between them"
- **Flaw:** Read against the actual file, `## Procedure` step 5 ends at `skills/dev/meeting-prep/SKILL.md:75`, but `## Tips` doesn't start until line 88 — and lines 77-86 contain a **third, complete section**: `## Triggering 15 min before each meeting` (a heading + fenced `cron:` YAML example), not mentioned anywhere in Phase 2. The plan's Requirements state "Placed after `## Procedure` and before `## Tips`" (`phase-02...md:34`) and its Implementation Steps frame lines 75→88 as if contiguous ("Locate the end of Procedure... and the start of Tips... Insert this section between them"), which reads as "insert immediately after 75," landing the new section between `## Procedure` and `## Triggering...` rather than adjacent to `## Tips` as the Requirements literally promise.
- **Failure scenario:** An implementer (human or agent) following the step literally inserts `## Memory discipline` right after line 75/76, splitting the natural "Procedure → Triggering (the scheduling detail for that same procedure) → Tips" reading flow with an unrelated discipline note wedged in between. This technically satisfies "before Tips" but violates the plan's stated goal of "keeps the read-first output/procedure flow intact" (`phase-02...md:34`) since it now interrupts the Procedure→Triggering continuity instead. At minimum the plan's own line citations (`75`, `88`) are factually incomplete — they omit an entire heading that any diff/grep-verification step (`phase-02...md:72`, "Verify no other section changed") would immediately surface, but the plan didn't catch it at authoring time.
- **Evidence:** `skills/dev/meeting-prep/SKILL.md:57` (`## Procedure`), `:75` (step 5, "5. **Attach**..."), `:77` (`## Triggering 15 min before each meeting`), `:86` (end of that section's fenced block), `:88` (`## Tips`).
- **Suggested fix:** Explicitly state the insertion point is "after `## Triggering...` (line 86), before `## Tips` (line 88)" — not "after Procedure line 75" — so the new section sits adjacent to Tips as actually intended, and re-verify the diff shows the discipline section directly preceding `## Tips`.

## Finding 6: Parallel Phase 1 + Phase 2 both need to flip status in the shared `plan.md` Phases table — an already-known, unaddressed race

- **Severity:** Medium
- **Location:** plan.md, "Dependencies" / "Intra-plan dependency matrix" (lines 59-69); "Execution: Phases 1 and 2 run in parallel (group A — disjoint files, no shared edits)."
- **Flaw:** "No shared edits" is true only for the *content* files (`production.yaml` vs `meeting-prep/SKILL.md`). It is not true for `plan.md` itself: both Phase 1 and Phase 2 must update their row in the shared `## Phases` status table (`plan.md:51-55`, currently both "Pending") when each completes, and Phase 3 is `blockedBy: [1, 2]` — something must read/flip both rows to unblock Phase 3. This is the *exact* race class this same repo's own sibling plan already identified and left unresolved: `plans/260703-0347-hermes-coding-agent-delegation-skill/plan.md:83` documents finding "#14: Parallel Phase 1 + Phase 2 both flip status in the shared `plan.md` Phases table — no write-ownership protocol for concurrent sessions," rejected there only with "not a defect this specific feature plan introduced... flagging for visibility... a fix would belong in the planning-tool's own conventions" (`:85`). This current plan cites that same sibling plan as its "Style precedent" (`phase-01...md:14`) but never re-surfaces or accounts for the identical structural risk it inherits by also running two phases in parallel with a shared status table.
- **Failure scenario:** Two concurrent sessions (or one session doing rapid sequential-but-unsynchronized writes) both read `plan.md`, each flips only their own row, and one write clobbers the other's edit (classic read-modify-write race on a shared file) if the tail-end save isn't row-scoped — Phase 3 could start prematurely (if only one row shows Done and the checker doesn't verify the second) or never unblock (if both writes race and one flip is lost).
- **Evidence:** `plans/260703-0347-hermes-coding-agent-delegation-skill/plan.md:83,85` (the identical finding, explicitly unresolved / deferred as a tooling-level gap); this plan's own `plan.md:61-69` sets up the same parallel-group-A-into-shared-plan.md structure without any mention of the precedent finding.
- **Suggested fix:** At minimum, note the known race in this plan's Risk/Red-Team section (matching the transparency the sibling plan already modeled) rather than silently repeating the same structural setup without acknowledgment. If the planning tool has since added row-scoped writes, cite that; otherwise, serialize Phase 1 and Phase 2 status updates or have a single session own both status flips.
