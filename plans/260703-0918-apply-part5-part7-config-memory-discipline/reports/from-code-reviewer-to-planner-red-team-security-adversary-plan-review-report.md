# Red Team Review — Security Adversary — Plan: apply-part5-part7-config-memory-discipline

Reviewer role: Fact Checker + Contract Verifier. All findings grep/read-verified against the live repo, not just plan prose.

## Finding 1: `memory_audit.suspicious_patterns` is a fabricated detection mechanism, same category of invention the plan claims to have cut

- **Severity:** Critical
- **Location:** Phase 1, section "Insert the `memory_audit:` block" (implementation step 3, lines 104-133)
- **Flaw:** The plan explicitly prides itself on cutting `curator`'s fabricated scoring weights (RT-1: "Do not fabricate a scoring formula... cut them"). It does not apply that same discipline to `memory_audit`. The block ships:
  ```yaml
  suspicious_patterns:
    - 'TODO|task|progress'
    - 'temporary|temp|WIP'
  ```
  cited to `part7:28-30,129`. But `part7-memory-system.md:27-30` and `:129` are plain-English prose in a two-column table ("Save task progress to memory" / "Use session_search to recall") — there is no regex, no pattern-matching spec, no mention that `/journey` or any Hermes feature does automated substring/regex classification of memory content. The regex patterns are invented by this plan, not "traced to a cited guide line" as Phase 1's own Requirements section demands ("Every field traces to a cited guide line... No invented scoring floats").
- **Failure scenario:** An operator deploys this config trusting the inline citation, believes Hermes actually flags entries matching these patterns, and stops manually reading `/journey` output closely — assuming the "suspicious_patterns" filter already caught anti-pattern entries. If the underlying engine has no `suspicious_patterns` field (untested — see Finding 2), the field is silently ignored and zero entries are ever flagged, producing false operator confidence that memory hygiene is automated when it is not.
- **Evidence:** `part7-memory-system.md:27-30` (prose table, no regex spec), `part7-memory-system.md:125-133` (anti-patterns table, no regex spec). Plan's own stated bar for Phase 1 (`phase-01-...md:40`): "Every field traces to a cited guide line ... No invented scoring floats" — `suspicious_patterns` violates this bar exactly as RT-1 found for curator weights, but RT-1's mitigation was never extended to this block.
- **Suggested fix:** Cut `suspicious_patterns` entirely (mirror the RT-1 treatment given to curator weights), replace with a comment naming the anti-pattern categories from `part7:125-133` without asserting a regex mechanism exists.

## Finding 2: `/journey` is documented as an interactive human-driven review tool, not a headless job a scheduler can invoke to emit a report

- **Severity:** Critical
- **Location:** Phase 1, section "Architecture / Data Flow" (lines 46-57) and "Key Insights" (line 31)
- **Flaw:** The entire `memory_audit:` block's premise is that a cron-like `schedule` field triggers "a `/journey` review pass" that autonomously "REPORTS stale/suspicious facts" via Telegram DM with no human in the loop. But `part7-memory-system.md:116-123` describes `/journey` as: "a timeline of every memory and skill Hermes has accumulated; **edit or delete any entry in place**. **The desktop app renders it** as a playable memory graph... Do a `/journey` pruning pass **monthly**" — i.e., a human opens the desktop app or runs the command themselves and reviews/edits interactively. Nothing in Part 7 (or anywhere else in the guide, per grep) describes a headless/batch invocation of `/journey` that can run unattended on a cron schedule and push a structured notification.
- **Failure scenario:** The config ships `memory_audit.schedule: "0 2 1 * *"` implying an automated monthly job fires without operator action. If the underlying engine has no headless `/journey` mode (plausible, since the guide only shows interactive usage), this entire block is dead config — the "monthly memory hygiene" this phase's whole rationale rests on never actually runs, and the operator has no monthly nudge unless they remember to open `/journey` themselves, exactly the gap this phase claims to close.
- **Evidence:** `part7-memory-system.md:116-123` — no CLI flag, no `--report`/`--json` mode, no cron-compatible invocation shown anywhere in the source chapter.
- **Suggested fix:** Either find/cite a guide passage describing a non-interactive `/journey` invocation (none currently exists per grep), or reframe `memory_audit:` as a **scheduled reminder/notification** ("time to run `/journey` — it's been 30 days") rather than implying an automated report is generated, which is the accurate claim the guide supports.

## Finding 3: security-hardened.yaml — the profile most exposed to untrusted/injected content — is explicitly excluded from the memory hygiene rollout, inconsistent with its own stated threat model

- **Severity:** High
- **Location:** Phase 1, "Overview" (line 26) and "Security Considerations" (lines 166-170); plan.md line 46-47
- **Flaw:** `templates/config/security-hardened.yaml:1-12` states its purpose in the header comment: "For operators whose agent reads untrusted content (public bots, webhook-driven flows, email inbox, GitHub PR comments)... Periodic security crons." This is precisely the deployment profile where a prompt-injection payload persisted into memory (a poisoned "user preference" or "environment fact" written via social-engineering of the agent) does the most damage, since Part 7 itself warns "a wrong memory gets injected into every future session and compounds" (`part7:123`). Yet the plan deliberately ships `curator:`/`memory_audit:` only to `production.yaml`, explicitly stating "No `security-hardened.yaml` change (admin config, not conversation capability)" — the exact same "admin config" characterization used to justify adding the blocks *without* approval-gating to `production.yaml` (Phase 1 Security Considerations line 168). The reasoning is applied inconsistently: same admin-config class, opposite outcome (added vs. explicitly withheld) between two nearly-identical templates.
- **Failure scenario:** An operator running the hardened profile for a public-facing bot (the file's stated use case) gets `security-hardened.yaml`'s existing `cron:` block (`security-hardened.yaml:110-120`: mcp-audit, bypass-audit, secret-rotate, log-sweep, disk-watchdog) — a security-conscious cron surface — but no equivalent memory-hygiene job, even though `security.secrets.memory_write_redaction: true` is already present there (`security-hardened.yaml:87`) showing the file's authors already care about memory-tier risk. The one profile designed for adversarial input is left without the review cadence this whole plan exists to add.
- **Evidence:** `security-hardened.yaml:1-12` (header, "reads untrusted content"), `:87` (`memory_write_redaction: true` already present), `:110-120` (existing security `cron:` block that memory_audit would naturally extend).
- **Suggested fix:** At minimum, flag this exclusion as an open question to the user rather than a silent unilateral decision — the plan currently states it as settled fact with no red-team line item covering it (RT-1 through RT-8 never address security-hardened.yaml).

## Finding 4: Phase 2's file-location claim is factually wrong — an entire intervening section is omitted from the plan's map of the target file

- **Severity:** High
- **Location:** Phase 2, section "Implementation Steps" step 1 (line 48) and "Overview" (line 34, "Placed after `## Procedure` and before `## Tips`")
- **Flaw:** The plan states: "Locate the end of `## Procedure` (line 75, step 5) and the start of `## Tips` (line 88)" and instructs inserting the new section "between them." Reading the actual file, `## Procedure` ends at line 75, but `## Tips` does NOT start at line 88 as the immediate next section — there is an entire un-mentioned section in between: `## Triggering 15 min before each meeting` (`skills/dev/meeting-prep/SKILL.md:77-86`), which contains a `cron:` YAML snippet (lines 79-86). The plan's Context Links, Related Code Files, and Implementation Steps never reference this section at all.
- **Failure scenario:** An implementer following the plan literally either (a) inserts `## Memory discipline` between `## Procedure` and `## Triggering...`, splitting the procedure from its own triggering-mechanism documentation in a way the plan never intended or reviewed, or (b) inserts it between `## Triggering...` and `## Tips` without the plan ever having accounted for that placement decision. Either way, the plan's Success Criteria claim "Frontmatter, `toolsets:`, output shape, and the 5 procedure steps are byte-identical to before" (line 86) is verifiable, but the claim that this is a clean 2-section insertion is not — the plan's structural model of the file is incomplete, which is exactly the kind of `file:line` fact a Fact Checker pass is meant to catch before implementation.
- **Evidence:** `skills/dev/meeting-prep/SKILL.md:77-86` (the missing section, confirmed via direct read) vs. plan's claim at `phase-02-...md:48` that line 88 (`## Tips`) immediately follows line 75.
- **Suggested fix:** Correct the plan to explicitly place the new section either before or after `## Triggering...` (recommend after, immediately preceding `## Tips`, so cron/triggering mechanics stay adjacent to the procedure they trigger), and update the line-number citations.

## Finding 5: `pinned_skills` protects exactly the skills Curator's default scope already excludes, per the guide's own text

- **Severity:** Medium
- **Location:** Phase 1, implementation step 2 field table (lines 70-77), field `pinned_skills`
- **Flaw:** `part5-creating-skills.md:141` states Curator "**Focuses on agent-created skills** first, not bundled/vendor skills." The four skills the plan pins — `coding-agent-delegate`, `nightly-backup`, `audit-mcp`, `rotate-secrets` — are all bundled/vendor skills shipped in this very repo (`skills/dev/coding-agent-delegate/`, `skills/ops/nightly-backup/`, `skills/security/audit-mcp/`, `skills/security/rotate-secrets/`, all confirmed present via `ls`/grep and all listed in `skills/README.md:32-42` catalog). Per the guide's own description of Curator's scope, these bundled skills would not be auto-archive candidates in the first place, making the `pinned_skills` list either inert (protecting against a risk that doesn't apply to them) or a sign the config's author misunderstands what Curator actually targets.
- **Failure scenario:** An operator reads `pinned_skills` and believes these four are specially protected from Curator, when in fact (per the guide) they were never at risk — meanwhile any *actually* agent-created skill the operator cares about (which the guide says IS in scope for auto-archival) goes unpinned because the template's example list gave a false sense that "the important ones are already covered."
- **Evidence:** `part5-creating-skills.md:141` ("Focuses on agent-created skills first, not bundled/vendor skills"); `skills/README.md:32-42` (catalog confirming all four pinned names are bundled skills, not agent-created ones).
- **Suggested fix:** Either cite the guide line justifying pinning bundled skills anyway (defense-in-depth against a future Curator version widening scope), or replace the example with a comment clarifying `pinned_skills` is a template placeholder for agent-created runbooks the operator will actually create, not these four bundled skills.

## Finding 6: No verified contract for how `curator.schedule`/`memory_audit.schedule` are actually consumed — assumed to share the `cron:` engine, but structurally are not part of the `cron:` list at all

- **Severity:** Medium
- **Location:** Phase 1, "Architecture / Data Flow" (lines 46-54); Risk Assessment row 1 (line 161)
- **Flaw:** The architecture diagram asserts: "Scheduler (Hermes cron engine, **same one that drives `cron:` block**)" runs `curator.schedule` and `memory_audit.schedule`. But structurally, `templates/config/production.yaml:289-294` shows `cron:` is a YAML list of maps (`{ name, schedule, task, notify }`), while the new `curator:`/`memory_audit:` blocks are independent top-level keys with their own scalar `schedule:` field, not entries appended to that list. This repo ships no runtime/engine code (config-templates-and-docs only) to verify this claim against, so "same scheduler consumes it" is asserted, not confirmed. The Risk Assessment table's own line 1 half-concedes this ("Curator engine ignores unknown keys, block is inert... harmless if a field is unread") but frames a potentially total-non-functionality risk as "Medium likelihood / Low impact," which undersells it: if the schema is wrong, the entire phase ships zero behavioral effect, only documentation-as-config.
- **Failure scenario:** Operator enables this template as-is expecting weekly curator runs and monthly memory audits per the comments; if the real Hermes engine expects `curator:`/`memory_audit:` keys nested under `cron:` (or an entirely different schema, e.g., top-level `schedules:` list), both blocks are silently inert and no hygiene job ever fires — while `git diff` and YAML-parse checks (the plan's only validation gates, step 4-5) both pass, giving false confidence the feature works.
- **Evidence:** `templates/config/production.yaml:289-294` (actual `cron:` list shape) vs. plan's proposed blocks (`phase-01-...md:82-133`, top-level scalar `schedule:` fields, not list entries). No engine/runtime code exists in this repo to confirm either schema is correct (confirmed via repo-wide search — this is a docs/templates-only repo).
- **Suggested fix:** Downgrade the "Impact: Low" rating for this risk to at least Medium, and/or add a note recommending the operator confirm the actual Hermes version's config schema (`hermes curator status` / `hermes config validate`, already mentioned in plan.md's Next Steps) before relying on this template in production — the plan should not present schedule wiring as settled when it's unverifiable from this repo alone.
