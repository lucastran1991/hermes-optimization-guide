# Red-Team Review: CCS/Claude-Code Delegation Timeout Fix Plan

Verified against real source at `/home/ubuntu/workspace/hermes-agent` (tools/terminal_tool.py,
tools/process_registry.py) and the live host (`templates/systemd/hermes.service`,
`/home/hermes/.hermes/logs/`). All findings are grep/read-verified, not inferred from plan prose.

## Finding 1: "Background mode survives Hermes restarts" is false — child shares the gateway's cgroup and dies with it

- **Severity:** Critical
- **Location:** Phase 1, section "Requirements" item 2 (background command template + comment "Redirecting to a log file ... survives Hermes restarts"); Overview's framing of `background=true` as *the* fix for the exact restart scenario cited in this review's prompt.
- **Flaw:** The plan claims redirecting stream-json output to a file makes the delegation "survive Hermes restarts." It does not verify what happens to the *process itself* (only the log file) when the gateway is killed by systemd during a restart — and the process does not survive.
- **Failure scenario:** `templates/systemd/hermes.service` has no `KillMode=` override (`grep -i kill` against `systemctl cat hermes.service` returns nothing but comments/`ExecReload`), so systemd's default `KillMode=control-group` applies: on `stop`/`restart`, systemd signals **all processes in the unit's cgroup**, not just the main PID. The unit's own comment at line 81 already states this explicitly: *"a delegated coding-agent sub-session (coding-agent-delegate skill, same UID/cgroup as this service)"*. `process_registry.spawn_local`/`spawn_via_env` launch the child with `start_new_session=True` (setsid) — this detaches the process *group/session* (relevant for signal-propagation-by-pgid and terminal ownership) but does **not** move the child to a different cgroup, so it remains inside `hermes.service`'s cgroup and is killed exactly like the parent when systemd tears the unit down (this project's own `plans/260703-1738-.../phase-02-deploy-p0-systemd-fix.md:20,81` already documents restarts landing mid-tool-call can take up to the default `TimeoutStopSec=90s` before SIGKILL — confirmed still unset on the live unit). So the exact scenario in the prompt (gateway restart mid-delegation) kills the backgrounded `ccs`/`claude` child too — the delegation is lost just as completely as today's `exit_code 124`, only silently (no error surfaces to the caller at all, since the caller/gateway process is also dead).
  Even in the counterfactual best case (child somehow survived), `recover_from_checkpoint()` (`tools/process_registry.py:1733-1826`) marks recovered sessions `detached=True` with the comment *"Can't read output, but can report status + kill"* (line 1793) — the in-memory `output_buffer` is empty on a fresh process object, and `notify_on_complete`'s payload is built from `session.output_buffer[-2000:]` (`tools/process_registry.py:1041`), so even a "successful" recovery delivers an empty-content completion notification. The redirected log file on disk is the only thing that actually survives — but nothing in the plan tells the agent to fall back to reading that file directly when `poll`/`wait` come back for a recovered/detached session with no buffer.
- **Evidence:** `templates/systemd/hermes.service` (no `KillMode=`/`TimeoutStopSec=` directive; line 81 comment: "same UID/cgroup as this service"); `tools/process_registry.py:754,822` (`start_new_session=True`, `id=f"proc_{uuid.uuid4().hex[:12]}"`); `tools/process_registry.py:1793` (`detached=True, # Can't read output...`); `tools/process_registry.py:1039-1051` (`notify_on_complete` payload sourced from `session.output_buffer`); `plans/260703-1738-fix-urgent-hermes-delegation-issues/phase-02-deploy-p0-systemd-fix.md:20,44,81` (documents the up-to-90s SIGKILL restart behavior on this exact unit).
- **Suggested fix:** Either (a) run delegated background commands in a separate systemd scope/cgroup (`systemd-run --scope --user` is not applicable for a system service without more plumbing — realistically, document that a restart during a delegation is a **known loss window** and add a recovery instruction: on gateway start, `tail`/re-read the redirected `.jsonl` log for any `delegation-*.jsonl` newer than the last known-good checkpoint and surface it to the user), or (b) explicitly scope this plan's claim down to "reduces loss window from the full task duration (today) to the restart-window duration (rare)" instead of asserting full survival — and add a Phase 3 check that intentionally restarts the gateway mid-delegation to observe actual behavior instead of assuming it.

## Finding 2: `process(action="wait")` is not unbounded — it silently caps at `TERMINAL_TIMEOUT` (180s default), the same number this plan exists to escape

- **Severity:** Critical
- **Location:** Phase 1, "Key Insights" bullet 3 ("`wait` = block until the backgrounded process exits ... safe to use here"); Requirements item 5 ("`process(action=\"wait\", session_id=...)` when the caller can afford to block").
- **Flaw:** The plan documents `wait` as blocking until completion. The real implementation clamps the wait window to `TERMINAL_TIMEOUT` (env var, default 180s) and returns `status: "timeout"` (not a completed result) once that elapses — regardless of what `timeout=` value the caller passes.
- **Failure scenario:** `ProcessRegistry.wait()` (`tools/process_registry.py:1284-1369`) computes `default_timeout = int(os.getenv("TERMINAL_TIMEOUT", "180"))` (line 1300) and sets `max_timeout = default_timeout` (line 1303); if the caller requests a longer `timeout`, it is silently clamped (lines 1307-1312, `timeout_note` explains the clamp but the caller must read that field to notice). The loop breaks after `effective_timeout` seconds and returns `{"status": "timeout", ...}` (lines 1360-1369) — the process is *not* killed, but the agent gets back a non-completion result identical in spirit to today's bug: no final output, just "still running." A `/ck:brainstorm`/`/ck:cook` run (the exact tasks this plan targets) routinely exceeds 180s. If the delegating agent follows Phase 1's documented pattern literally — call `wait` once "when it can afford to block" — it gets a `timeout` status after 180s and, per the plan's docs, has no instructed retry/loop behavior. Nothing in Phase 1 tells the agent to loop `wait` calls or fall back to `poll`+sleep when `status == "timeout"`.
  This also undermines Phase 3: its smoke test deliberately picks a *short* task ("`/ck:scout` against a small repo — avoid a full `/ck:brainstorm --hard` ... to keep cost/time down", Implementation Step 6: "Do not run a full `/ck:brainstorm --hard`") specifically so it finishes fast — which means the smoke test can never exercise the >180s path where `wait`'s real ceiling matters, so this bug would ship undetected even after Phase 3 "passes."
- **Evidence:** `tools/process_registry.py:1300-1314` (clamp logic), `:1360-1369` (`timeout` status return), confirmed no `TERMINAL_TIMEOUT` override in `/home/hermes/.hermes/.env` (empty grep, matches plan's own claim in Phase 1 Key Insights) so 180s is what's actually active on this host today.
- **Suggested fix:** Phase 1 Requirements item 5 must document `wait` as a bounded poll primitive (loop `wait(timeout=N)` until `status == "exited"`, treating `"timeout"` as "still running, call again" — not an error), and Phase 3 must add an explicit test with a task that runs longer than `TERMINAL_TIMEOUT` (e.g. a deliberately slow but cheap sleep-based fixture, not a real brainstorm run) to prove the loop pattern actually works end-to-end.

## Finding 3: `delegation-<run-id>.jsonl` has no defined uniqueness scheme — structurally cannot use the tool's own unique session_id, and `parallel=N` fan-out is a direct collision path

- **Severity:** High
- **Location:** Phase 1, Requirements item 2 (the `terminal(...)` command template with `> /home/hermes/.hermes/logs/delegation-<run-id>.jsonl`); Success Criteria item 5 ("`parallel=3` example updated to use background mode").
- **Flaw:** `<run-id>` is never defined — no generation rule, no example, no reference to an existing convention. Worse, it *cannot* be the terminal tool's real unique ID: `terminal_tool()`'s background call assigns `session_id` via `uuid.uuid4()` inside `spawn_local`/`spawn_via_env` (`tools/process_registry.py:682,822`) — i.e., *after* the shell `command` string (which must already contain the literal redirect path) has been constructed by the calling agent. The redirect target must be chosen before the tool call returns a real unique ID, so `<run-id>` is necessarily some ad-hoc value the delegating LLM invents on the spot (timestamp, counter, etc.), with zero enforcement.
- **Failure scenario:** `parallel=3` (the exact scenario Phase 1 explicitly requires updating per Success Criteria item 5, and the exact scenario in the failure-mode prompt) issues 3 concurrent `terminal(background=true, ...)` calls within the same LLM turn. If the agent derives `<run-id>` from a coarse timestamp (e.g. second-granularity `date +%s`, a very plausible naive choice with nothing in the skill telling it otherwise), two or three subtasks can compute the identical filename. Shell `>` truncates on open — the second process's redirect silently truncates whatever the first has already written, and both processes then interleave raw bytes into the same fd, corrupting both jsonl transcripts with no error surfaced to either delegation (redirection collisions are silent at the shell level).
  Note the skill already has a working disambiguator nearby it could reuse — the existing `parallel=3` worktree example uses `devin/claude-code-<ts>-subtask-N` (`skills/dev/coding-agent-delegate/SKILL.md:182`) — but Phase 1 never instructs reusing/extending that pattern for the log path, so there's no guarantee the rewritten example does the safe thing.
- **Evidence:** `tools/process_registry.py:682,822` (session_id generated post-spawn via `uuid.uuid4()`); `skills/dev/coding-agent-delegate/SKILL.md:182` (existing `<ts>-subtask-N` convention, not referenced by Phase 1's requirements for the log path).
- **Suggested fix:** Mandate the log filename use the same `<ts>-subtask-N` (or a `mktemp`-style unique suffix, e.g. `$$`-pid + nanosecond timestamp) already used for worktree naming in the fan-out example, and add a Success Criteria line asserting the `parallel=N` rewritten example produces N distinct paths.

## Finding 4: No log rotation for the new artifact, on a host where /home/hermes is shared with the production gateway's own state

- **Severity:** High
- **Location:** Phase 1, "Risk Assessment" ("log file ... no rotation is defined ... out of scope for this phase").
- **Flaw:** Deferring log rotation as out-of-scope is presented as low-risk ("flag under Unresolved Questions for a follow-up if it becomes noisy"), but there is currently **no** rotation mechanism anywhere in this repo for `/home/hermes/.hermes/logs/` at all (not even for the pre-existing `agent.log`/`gateway.log`), and the new artifact is materially larger per run than anything currently written there.
- **Failure scenario:** `--output-format stream-json --verbose --include-partial-messages` (mandated by Phase 1's template) emits one JSON line per streamed token/partial event, not just final output — this is verbose by design and, for a multi-agent `/ck:brainstorm --hard` or `/ck:cook --parallel` run (exactly the tasks this fix targets), can produce jsonl files far larger than the `--output-format json` single-shot output used today. `MAX_OUTPUT_CHARS = 200_000` (`tools/process_registry.py:58`) only bounds the in-memory buffer used for `poll`/`notify` payloads — it does **not** bound the actual bytes written to the redirected file on disk, which is unbounded shell redirection. `ReadWritePaths=/home/hermes/.hermes /home/hermes/.ccs /home/hermes/.claude /tmp` (hermes.service:54) is also the writable path set the gateway itself depends on for its own state/session data — if `/home/hermes` fills up (single filesystem, no separate mount evidenced), the production bot's own operation is at risk, not just the delegation feature. Confirmed live: no logrotate config exists anywhere under `templates/` (`grep -rn logrotate` → no hits), and `/home/hermes/.hermes/logs/` currently holds hand-grown logs (`agent.log` 491KB, `gateway.log` 119KB as of this review) with no existing cleanup cron either.
- **Evidence:** `tools/process_registry.py:58` (`MAX_OUTPUT_CHARS` is an in-memory cap only); `templates/systemd/hermes.service:54` (shared `ReadWritePaths`); live `sudo -n -u hermes ls -la /home/hermes/.hermes/logs/` (no rotation artifacts, files already hundreds of KB); repo-wide grep for `logrotate` → zero results.
- **Suggested fix:** At minimum, add a size/age cap directly in the command template (e.g. pipe through a rotation-aware wrapper, or a `find ... -mtime +N -delete` cron for `delegation-*.jsonl`) before shipping — "out of scope" is acceptable for a first pass only if paired with a hard success-criterion that these files are cleaned up, not merely "flagged" for later.

## Finding 5: Phase 3's "no orphaned background sessions" success criterion has no automated teardown — pure manual discipline on a live production host

- **Severity:** Medium
- **Location:** Phase 3, "Non-functional" requirement and Success Criteria ("No orphaned background process left running after the phase closes").
- **Flaw:** The only stated mitigation is a to-do-style instruction ("`process(action=\"kill\")` or let it complete naturally before ending this phase") with no scripted/forced cleanup step, no `finally`-equivalent, and no verification command listed to confirm cleanup actually happened.
- **Failure scenario:** If the live smoke test hangs (e.g. the CLI blocks on an unexpected interactive prompt, a network stall inside `claude`, or the human/agent running Phase 3 is interrupted before reaching the kill/wait step), the phase has no forcing function to catch it — it depends entirely on the operator remembering. This is happening on the actual production hermes host serving real Telegram traffic (Phase 3's own Risk Assessment acknowledges this), so a forgotten orphan silently consumes a process slot (`MAX_PROCESSES = 64`, `tools/process_registry.py:60`) and CPU/output-buffer memory indefinitely (`FINISHED_TTL_SECONDS` only prunes *finished* sessions, not running ones) until someone notices.
- **Evidence:** Phase 3 file, "Non-functional" and Success Criteria sections (no scripted verification step); `tools/process_registry.py:60` (`MAX_PROCESSES = 64` — finite budget shared with all other terminal/process usage on the host).
- **Suggested fix:** Add an explicit final verification command to Phase 3's Implementation Steps — e.g. `process(action="list")` (or equivalent) confirming zero running sessions matching the smoke-test's command pattern before marking the phase done — as a hard gate, not a checklist reminder.

## Finding 6: Phase 3's smoke test doesn't specify testing the real skill-invocation path — may validate CLI reachability, not the actual bug being fixed

- **Severity:** Medium
- **Location:** Phase 3, Requirements item 1 and Implementation Step 1 (`sudo -u hermes env PATH=... ccs ccs-hermes -p "echo ok" --output-format json`); Requirements item 7 (the `toolsets:` gating question).
- **Flaw:** Phase 3's steps are written as raw shell commands run directly as the `hermes` user, not as an actual Telegram message triggering the `coding-agent-delegate` skill through Hermes' own tool-dispatch layer. Requirement 7 explicitly needs to know whether the `coding-agent-delegate` skill session has `terminal`/`process` tool access "before vs. after adding `terminal` to its `toolsets:` list" — but nothing in Phase 3 describes driving that through the skill/agent path (vs. a bare CLI call, which bypasses skill-level tool gating entirely by construction, since it isn't going through Hermes at all).
- **Failure scenario:** Phase 3 could report "PASS" based entirely on manual CLI invocation of `ccs`/`claude` (which always works regardless of any skill's `toolsets:` frontmatter, since that path never touches skill-level tool access), while the actual agent-driven path — a user's Telegram message invoking `/coding-agent-delegate`, which is what actually exercises `terminal(background=true)` from *within* the skill's granted toolset — is never exercised, leaving the real fix unverified end-to-end.
- **Evidence:** Phase 3 file, Implementation Steps 1-2 (all raw `sudo -u hermes ... ccs/claude` invocations, no described Telegram/gateway-driven trigger); Requirement item 7's own admission it may be "practically impossible to isolate."
- **Suggested fix:** Add one explicit step that triggers the delegation via an actual Telegram message to the bot (or the gateway's programmatic equivalent) invoking `/coding-agent-delegate`, not just a bare shell command, so the smoke test actually exercises the skill-dispatch → `terminal(background=true)` path this plan changes.

- **Severity:** Medium
- **Location:** Phase 1, "Unresolved Questions" item 1; Phase 3, Requirements item 7 (carries the same question forward as something to test live).
- **Flaw:** The plan treats "does `toolsets:` frontmatter gate runtime tool access?" as something requiring live-host experimentation. A direct read of the actual skill/frontmatter parser answers it with high confidence without touching the live host at all.

If Phase 3 falls back to its own documented escape hatch ("if practically impossible to isolate, downgrade to documented as unresolved"), the plan ships still calling this unresolved when it isn't.
- **Evidence:** `agent/skill_utils.py` frontmatter accessors only read `platforms`, `environments`, `metadata`, `name`, `description` (lines 183, 290, 543, 577, 643, 712) — no `toolsets` key is read anywhere. Repo-wide `grep -n '"toolsets"'` across `agent/*.py` returns zero hits for a literal dict-key lookup (the only "toolsets" hits in the whole `hermes-agent` tree are `requires_toolsets`/`fallback_for_toolsets`/`enabled_toolsets`/`available_toolsets` — a *different*, agent-level mechanism unrelated to a skill's own `toolsets:` frontmatter field). This confirms `toolsets:` in `coding-agent-delegate/SKILL.md` is decorative prose, not code-enforced — consistent with the precedent the plan itself already cites (`delegation.routing` being LLM-interpreted, not code-enforced).
- **Suggested fix:** Resolve Phase 1's Unresolved Question 1 now via this citation instead of carrying it into Phase 3; redirect the freed live-verification budget toward Findings 1/2 (restart survival, `wait` timeout looping) which are not resolvable by static grep alone.
