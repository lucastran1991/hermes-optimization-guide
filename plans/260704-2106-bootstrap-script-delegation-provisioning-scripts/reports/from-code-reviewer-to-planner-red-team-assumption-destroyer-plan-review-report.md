# Red-Team Review: Bootstrap Script Delegation Provisioning Plan

Role: Assumption Destroyer / Scope Auditor. Verdict: plan has 3 Critical, 2 High, 2 Medium findings — do not execute Phase 6 or claim end-to-end success until findings 1-3 are resolved.

## Finding 1: Phase 6's "replace, don't duplicate" merge logic has no working safety net

- **Severity:** Critical
- **Location:** Phase 6, section "Idempotency" / Success Criteria
- **Flaw:** The plan's stated safety net for a duplicate top-level `delegation:` key is "YAML-validates; auto-restores `.bak` + `die`s on invalid YAML" (Success Criteria, Risk row "Duplicate `delegation:` top-level key (silent) | M×M"). But a duplicate top-level key is **not invalid YAML** — PyYAML's `safe_load` (the exact tool the plan specifies: "`python3 -c 'import yaml,sys; yaml.safe_load(...)'`", phase-06 Key Insights) silently keeps the last occurrence and does not raise.
- **Failure scenario:** If the "detect + replace existing block" text-merge logic has any bug (wrong regex boundary, blank-line miscount, re-run with `--force`-style overwrite skipped), the script appends a second `delegation:` block instead of replacing the first. The YAML-validate gate passes anyway (confirmed empirically below), the script reports success, and the live config silently runs with whichever block PyYAML/the Hermes gateway's own YAML loader happens to pick — masking the actual bug.
- **Evidence:** Verified directly:
  ```
  python3 -c "import yaml; print(yaml.safe_load('a: 1\nb: 2\na: 3\n'))"
  # -> {'a': 3, 'b': 2}   (no error, silently keeps last)
  ```
  Phase 6 lines describing the (only) validation step: "YAML-validate; on failure restore `.bak` + `die`." No secondary check (e.g. `grep -c '^delegation:'` count == 1) is specified anywhere in the phase.
- **Suggested fix:** Add an explicit post-merge assertion: `[ "$(grep -c '^delegation:' "$CONFIG")" -eq 1 ] || { restore .bak; die "duplicate delegation: block detected post-merge"; }` — don't rely on YAML validity as a duplicate-key detector, it structurally cannot catch this.

## Finding 2: Phase 6 needs a repo file but the plan's own top-level constraint mandates curl|sudo bash — no path resolution defined

- **Severity:** Critical
- **Location:** Phase 6, section "Related Code Files" / "Architecture"
- **Flaw:** Phase 6's script sources the `delegation:` block from `templates/config/production.yaml` ("Reads (repo): `templates/config/production.yaml`"). This requires a local git checkout. But `plan.md:26` states the hard design constraint: "every script stays safe under `curl | sudo bash` / non-interactive use" — under a pure `curl | sudo bash` invocation there is no local repo clone at all. Unlike `scripts/deploy-systemd-units.sh`, which the plan cites as the style to mirror (phase-02 Context Links) and which explicitly resolves `GUIDE_DIR="${GUIDE_DIR:-/opt/hermes-optimization-guide}"` plus a stale-clone guard (`deploy-systemd-units.sh:38,47-57`), Phase 6 defines **no** `GUIDE_DIR` variable, no fallback fetch-from-GitHub path, and no stale-clone check.
- **Failure scenario:** Operator runs `4-merge-delegation-config.sh --ccs-profile=ccs-hermes` per the documented usage. If invoked via the design's own `curl | sudo bash` pattern (or from any cwd other than a repo checkout), the script cannot find `templates/config/production.yaml` and either crashes with a bare "No such file" (bad UX, no `die` message written for this case) or — worse, if a stray/stale `/opt/hermes-optimization-guide` checkout is behind `origin/main` — silently merges a stale `delegation:` block, with no guard against that (Finding echoes `deploy-systemd-units.sh`'s own header comment: "a pre-fix template could get reinstalled").
- **Evidence:** `grep -n "GUIDE_DIR" plans/260704-2106.../phase-06*.md plan.md` returns zero matches. `deploy-systemd-units.sh:38,47-57` shows the exact pattern Phase 6 needed and skipped.
- **Suggested fix:** Either (a) explicitly scope Phase 6 as "run from a repo checkout only, not curl|bash-safe" and say so loudly (breaking the plan's own universal constraint, which should be a documented exception, not a silent gap), or (b) add the same `GUIDE_DIR` + stale-clone guard as `deploy-systemd-units.sh`.

## Finding 3: Plan does not fix — or even mention — the already-diagnosed hermes.service PATH bug that blocks the entire delegation flow at runtime

- **Severity:** Critical
- **Location:** Whole plan (no phase addresses it); most relevant to Phase 4/5 (ccs profile) and the plan's own closing goal ("manual real `/delegate_code` test")
- **Flaw:** `plans/reports/ck-debug-260704-1355-hermes-ccs-auth-profile-access-report.md` (dated the SAME day, ~7.5 hours before this plan was created) already root-caused why `harness: ccs` delegation fails on this exact host: `hermes.service`'s systemd sandbox uses the bare default PATH (`/usr/local/sbin:...:/snap/bin`) with **no** `Environment=PATH=` override, so `/home/hermes/.local/bin` (where `ccs`, `claude`, `ck` all live) is not on the service's PATH. Confirmed live: `journalctl -u hermes` shows `ccs: command not found` (exit 127); reproduced directly with `env -i HOME=... PATH=<systemd-default> ccs` → fails, `.../.local/bin/ccs` (absolute) → works. That report's own fix ("not applied"): add `Environment=PATH=/home/hermes/.local/bin:...` to `templates/systemd/hermes.service` + `deploy-systemd-units.sh`/restart.
- **Failure scenario:** This plan's Phases 2-6 successfully provision `gh auth`, an `ANTHROPIC_API_KEY`, a `ccs-hermes` profile, and wire `delegation:` into the live config — all necessary but NOT sufficient. When the mandatory human `/delegate_code` end-to-end test runs (the plan's own stated final gate, Phase 6 Next Steps), the coding-agent-delegate skill running inside `hermes.service` will invoke `ccs ccs-hermes -p "..."` by bare name and hit the exact same `command not found` (exit 127) already logged on 2026-07-03. All 6 phases' smoke-tests (`sudo -u hermes -i ccs ... `) will pass because `-i` sources `.bashrc` which prepends `~/.local/bin` — masking the bug exactly as the debug report warns ("interactive shell ≠ systemd sandbox"). The plan's own smoke-test methodology (every phase 2-5 verifies via `sudo -u hermes -i ...`) reproduces the false-positive this debug report explicitly flags.
- **Evidence:** `plans/reports/ck-debug-260704-1355-hermes-ccs-auth-profile-access-report.md:9-21` (PATH root cause + reproduction); `grep -n "PATH=" plans/260704-2106-bootstrap-script-delegation-provisioning-scripts/*.md` returns nothing — zero phase mentions this fix.
- **Suggested fix:** Add a Phase (or amend Phase 6) that applies the debug report's fix — `Environment=PATH=/home/hermes/.local/bin:...` in `templates/systemd/hermes.service`, deployed via the existing `scripts/deploy-systemd-units.sh` — before claiming the delegation flow is ready for the human `/delegate_code` test. Also change every phase's smoke-test to invoke via the SAME non-login mechanism `hermes.service` uses (e.g. `sudo -u hermes env -i HOME=/home/hermes PATH=<systemd-default> ccs ...`) instead of `sudo -u hermes -i`, or the smoke tests will keep passing while the service keeps failing.

## Finding 4: Phase 1's 6c idempotency guard silently breaks on re-run (unlike the section 6b pattern it claims to mirror)

- **Severity:** High
- **Location:** Phase 1, section "Architecture" (new section 6c)
- **Flaw:** Phase 1 states the new 6c ClaudeKit install command is: `sudo -u hermes bash -c 'command -v ck || npm install -g --prefix "$HOME/.local" claudekit-cli'` — no `PATH` export, no `-i`. Contrast section 6b (the pattern Phase 1 explicitly cites as the thing to mirror, "Pattern to mirror: `scripts/vps-bootstrap-oci.sh:123-155`"), which DOES `export PATH="$HOME/.local/bin:$PATH"` inside its `sudo -u hermes bash -c '...'` block before any `command -v` check (`vps-bootstrap-oci.sh:138-140`).
- **Failure scenario:** `ck` is installed to `~hermes/.local/bin/ck`. A bare (non-login, non-`-i`) `sudo -u hermes bash -c` does not source `.bashrc`/`.profile` and, unlike 6b, this 6c snippet never exports `~/.local/bin` into `PATH` itself. On any re-run of the bootstrap (the plan's own "Idempotency check" step 5), `command -v ck` will very plausibly return false again (PATH lacks `~/.local/bin`), so `npm install -g claudekit-cli` re-runs every single time — contradicting the plan's own Risk Assessment row ("Double-run reinstalls/errors | L×L | `command -v ck` ... guards make both sections idempotent") and Success Criteria ("Re-running bootstrap is a clean no-op").
- **Evidence:** `vps-bootstrap-oci.sh:138-140` (6b's explicit `export PATH=...` before its `command -v` checks) vs phase-01's 6c snippet (Architecture section, line 40) which has no such export.
- **Suggested fix:** Either export `PATH="$HOME/.local/bin:$PATH"` inside the 6c `bash -c` block exactly like 6b does, or guard on a file-existence check (`[ -x "$HOME/.local/bin/ck" ]`) instead of `command -v`.

## Finding 5: Phase 1 installs the entire CK harness into `~/.claude`, which sits outside `hermes.service`'s writable-paths allowlist

- **Severity:** High
- **Location:** Phase 1, section "Architecture" (6c/6d); cross-cutting with `templates/systemd/hermes.service`
- **Flaw:** `templates/systemd/hermes.service:41,45` sets `ProtectHome=read-only` and `ReadWritePaths=/home/hermes/.hermes /home/hermes/.ccs /tmp` — `/home/hermes/.claude` is not in that allowlist. Phase 1 installs ClaudeKit + `ck init --install-skills` + the skills venv entirely under `~/.claude` (confirmed live directory layout in `ck-debug-260704-1355...report.md:67`: `.claude/ (770) — agents/, skills/, commands/, plugins/, sessions/, settings.json`). Per `skills/dev/coding-agent-delegate/SKILL.md:48`, this `~/.claude/` presence is exactly what gates whether delegated `claude -p`/`ccs -p` subprocess invocations get the "full CK harness" — meaning those subprocess invocations, spawned from inside the `hermes.service` sandbox, read AND typically write session/state files there (settings, session transcripts, shell snapshots — visible in the live tree).
- **Failure scenario:** Bootstrap-time provisioning (root/`sudo -u hermes`, run manually or via the bootstrap script) happens OUTSIDE the systemd sandbox, so it succeeds and looks fine. At actual delegation runtime, `hermes.service`'s mount-namespace restrictions apply to every child process it forks (including the delegated `claude`/`ccs` subprocess), so any write attempt under `~/.claude` (new session file, updated settings.json, venv `__pycache__`) hits `EROFS`. Depending on how gracefully the CK harness degrades on a read-only home dir, this either silently drops session state or hard-fails the delegated call — a different failure mode from, but adjacent to, the PATH bug in Finding 3, and equally undocumented by this plan.
- **Evidence:** `templates/systemd/hermes.service:41,45` (no `.claude` in `ReadWritePaths`); `skills/dev/coding-agent-delegate/SKILL.md:48` (`~/.claude/` presence gates the harness); no phase in this plan mentions `ReadWritePaths` or proposes widening it for `.claude`.
- **Suggested fix:** Either add `/home/hermes/.claude` to `ReadWritePaths` in `templates/systemd/hermes.service` (redeployed via `deploy-systemd-units.sh`, same discipline as the PATH fix in Finding 3) or confirm/document that the CK harness only needs read access at delegation time and never writes under `~/.claude` — currently asserted nowhere.

## Finding 6: Phase 2's PAT-leak mitigation addresses the wrong layer

- **Severity:** Medium
- **Location:** Phase 2, "Security Considerations" / Risk Assessment
- **Flaw:** The stated mitigation is "Pass via STDIN (`--with-token`) not an arg to `gh`" — true, `gh`'s own argv never sees the token. But the wrapper script `0-gh-auth.sh` itself is invoked as `0-gh-auth.sh --token=<PAT>` (per its own documented usage), meaning the PAT sits in `0-gh-auth.sh`'s process arguments (visible via `ps aux`/`/proc/<pid>/cmdline` for the script's entire runtime) and, if the operator types it inline (a usage pattern the plan itself documents as the primary example), lands in the invoking shell's history — the exact class of leak the risk row claims is mitigated.
- **Failure scenario:** Any other process on the host (or a co-located delegated sub-session under the same UID, per the already-flagged same-UID exposure risk) running `ps aux` during the ~1-second window the script executes captures the raw PAT. Root's `~/.bash_history` on the OCI box also gets the literal token if run interactively rather than via `GH_TOKEN=... bash 0-gh-auth.sh`.
- **Evidence:** phase-02 lines: usage doc `--token=<PAT>` (primary path), Risk row "Token leaks via `ps`/shell history | M×M | Pass via STDIN ... not an arg to `gh`" — conflates the two distinct exposure surfaces (gh's argv vs the wrapper's argv).
- **Suggested fix:** Document `GH_TOKEN=<PAT> bash 0-gh-auth.sh` (already supported per the arg-resolution logic) as the ONLY recommended invocation in the header/usage text, and explicitly demote `--token=` to "not recommended, leaks via ps/history" rather than presenting both as equally safe.

## Finding 7: Phase 5 formalizes a shared-state leak that has already occurred manually on this host, without escalating its severity accordingly

- **Severity:** Medium
- **Location:** Phase 5, "Security Considerations"
- **Flaw:** Phase 5's own risk analysis (correctly) states the reuse-bridge script industrializes the known risk in `[[hermes-temporarily-shares-ubuntu-personal-claude-credential]]`. Live evidence confirms this is not hypothetical: `ck-debug-260704-1355...report.md:68,71` shows `/home/hermes/.ccs/instances/{lucas,ken}/` ALREADY present on the live host — the exact manual copy this script would automate has already happened. What's under-addressed: `.ccs/instances/<name>/` is a full CCS instance directory (session/auth state), not a single scoped API key — its blast radius on leak (e.g. via the already-flagged same-UID delegated-sub-session read exposure, `project_hermes-oauth-credential-exposed-to-delegated-subsessions.md`) is full personal-account impersonation, not "shared quota" as phrased in the Overview. The Risk Assessment table rates this H×H already, so the severity is acknowledged — but Phase 6's `4-merge-delegation-config.sh --ccs-profile=<name>` can point at EITHER Phase 4's dedicated profile or Phase 5's personal-bridge instance with the identical flag/mechanism, meaning nothing structurally prevents `ccs_profile: ken` from ending up wired into a "production" config template despite Phase 5 being internal-fork-only.
- **Failure scenario:** A future re-provisioning run (or a teammate following the numbered scripts literally) runs `3-ccs-reuse-bridge.sh --instance=ken --i-understand-the-risk` then `4-merge-delegation-config.sh --ccs-profile=ken` — nothing in Phase 6's script distinguishes "a dedicated bot profile name" from "a personal instance name" at merge time, so the fork-only warning from Phase 5 doesn't propagate into Phase 6's guardrails.
- **Evidence:** `ck-debug-260704-1355...report.md:68,71` (instances/{lucas,ken} already on host); Phase 5 Security Considerations (self-flagged H×H); Phase 6 Architecture (`--ccs-profile=<name>`, generic, no source-provenance check).
- **Suggested fix:** Have `4-merge-delegation-config.sh` `warn` (not block) when `--ccs-profile` doesn't match the constant `ccs-hermes` produced by Phase 4, nudging the operator to confirm they intend the bridge path.
