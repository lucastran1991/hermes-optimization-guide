`templates/systemd/hermes.service` sets `ProtectHome=read-only` (confirmed via `man systemd.exec`: read-only means the home tree stays visible/readable, only writes are blocked ‚Äî unlike `ProtectHome=true` which hides it entirely) and only grants `ReadWritePaths=/home/hermes/.hermes /home/hermes/.ccs /tmp` ‚Äî never `/home/hermes/.claude`. Because a delegated `claude -p ... --allowedTools "Read,Edit,Bash"` sub-session (per `skills/dev/coding-agent-delegate/SKILL.md:77`) runs as the same hermes UID as the top-level gateway process, Linux DAC "owner-only" file permissions provide zero protection against it ‚Äî the sub-session IS the owner. So whatever OAuth credential `claude auth login` persists under `/home/hermes/.claude*` is freely readable (and exfiltratable over the network ‚Äî no egress allowlist restricts a spawned Bash tool) by any delegated task, rogue or compromised.

**Why:** `skills/dev/coding-agent-delegate/SKILL.md:38` already states the governing principle ("Never pass writable production credentials into a sub-session"), but the `260703-1738-fix-urgent-hermes-delegation-issues` plan's Phase 3 provisions exactly such a credential without cross-referencing it, and its own Security Considerations text ("token is stored under the hermes home (owner-only)") is misleading ‚Äî "owner-only" defends against a different-UID attacker, not a same-UID delegated sub-session, which is the actual threat model here. Full detail: `plans/260703-1738-fix-urgent-hermes-delegation-issues/reports/from-code-reviewer-to-planner-red-team-security-adversary-plan-review-report.md` Finding 2. As of this writing the finding is reported but NOT yet remediated or explicitly risk-accepted by the user.

**How to apply:** If a future plan touches hermes's `claude auth`/OAuth setup, `ReadWritePaths`, or the `coding-agent-delegate` skill's tool allowlists, check whether this gap has been closed (e.g., via `CLAUDE_CONFIG_DIR` redirection isolating the credential from delegated sub-sessions, or an explicit accepted-risk note) before treating hermes's OAuth login as safe against a "delegated task goes rogue" scenario. Don't re-verify from scratch ‚Äî read the linked report first, then confirm current state against it (memories can go stale).
